# UX Heuristic Audit — ShipSync v0.2 ship detail editor

**Scope:** the v0.2 ship-detail editing surface — Particulars (profile + portrait), Captain (officer card), Player character (optional extension), plus the integrating refactor of `ShipDetail.svelte` and the small downstream changes in `AddShipDialog.svelte` and `Dashboard.svelte`. This is a delta audit on top of [`v0.1-ux-audit.md`](./v0.1-ux-audit.md): findings that already shipped at v0.1 are referenced rather than re-litigated.
**Date:** 2026-04-26
**Surfaces inventoried:** 6 components (3 new, 3 changed)

Components inventoried:

- `src/lib/features/ship/ProfileSection.svelte` — new
- `src/lib/features/ship/OfficerCard.svelte` — new (also slated to back the v0.3 officer grid)
- `src/lib/features/ship/PlayerCharacterSection.svelte` — new
- `src/lib/features/ship/ShipDetail.svelte` — refactored to host the three new sections, surface a portrait + last-saved time in the header, drop the dev-only roadmap stub
- `src/lib/features/fleet/AddShipDialog.svelte` — minor: a chartered ship's "portrait" upload now binds only to `playerCharacter.portraitImageId` (the ship-level portrait is added separately in Particulars)
- `src/routes/Dashboard.svelte` — minor: `overflow-x-auto` on the columns container so dense detail content doesn't collapse the rail or scene panel

The state-mutator cluster in `src/lib/state/workspace.svelte.js` (lines ~319–701) is in scope as the source of error-prevention behavior; it is not a UI surface.

## Findings by Heuristic

### 1. Visibility of System Status — PASS

- [x] Async actions show progress indicators
- [x] Loading states for disk-bound operations
- [x] Submit / mutation buttons disable + show progress while in-flight
- [x] Success is confirmed (toast / state change)
- [x] Long-running tasks expose progress beyond "loading"
- [x] Status indicators are reachable to assistive tech

**Evidence**

- `src/lib/features/ship/ShipDetail.svelte:130` — explicit "Last file save today at HH:mm" / "Last file save Apr 25, 4:18 PM" / "Not yet written to a file." line directly under the ship's name. This complements the autosave indicator in the top bar (which is global and crash-recovery-only) with a per-ship sense of how stale the on-disk copy is.
- `src/lib/state/workspace.svelte.js:319,408,431,614,646,683` — every commit appends a human-readable log line ("Hull max 60 → 65 on The Tideroar.", "Removed player-character extension from The Tideroar.", "Replaced player character portrait on The Tideroar."). The Captain's Log on the right rail is the in-line confirmation channel for inline edits, which gives every blur/change/toggle a visible "this happened" signal.
- `src/lib/features/ship/OfficerCard.svelte:121-148` — the segmented status control (Active / Stricken / Dead) tints the active option (brass / amber / crimson) so the officer's state is recognizable at a glance, not just on hover.
- `src/lib/features/ship/ShipDetail.svelte:113-118` — the header portrait (with `alt={`${ship.name} portrait`}`) is a non-text status cue that the file actually has the bundled image you uploaded.

**Findings**

- Note (carried from v0.1): `ImageUpload` doesn't surface a spinner while the `FileReader` runs. Triggers more often in v0.2 because there are now three places to upload portraits (ship, officer, PC). Still acceptable for the typical sub-MB image and `IMAGE_HARD_CAP_BYTES` covers the catastrophic case, so I'm leaving the verdict at PASS — but if upload feedback ever feels laggy, `ImageUpload` is the place to add a `busy` flag.

### 3. User Control and Freedom — PARTIAL

- [x] Every modal/sheet/dialog has a Close, Cancel, or back affordance — unchanged from v0.1
- [x] Multi-step flows have a way to step back without losing safe state — N/A (still no multi-step flows in v0.2)
- [x] Destructive actions are reversible (undo) or confirmable
- [x] Forms can be exited without losing entered data, or warn before discard — N/A for the inline editor (every commit goes through `commit()` and is undoable)
- [x] Navigation gestures (esc-to-close) behave as expected — unchanged

**Evidence**

- `src/lib/state/workspace.svelte.js:319,342,408,431,614,646,683` — every v0.2 mutator wraps its mutation in `commit(...)`, so the existing `Cmd/Ctrl+Z` and the toolbar undo button cover the whole new edit surface. There is no "destructive action" in v0.2 that escapes the undo stack.
- `src/lib/features/ship/ShipDetail.svelte:67-76` — `confirmAndRemove` still gates ship removal behind `window.confirm` and explicitly mentions saved files are untouched.
- `src/lib/features/ship/PlayerCharacterSection.svelte:54-62` — the PC toggle is a single click; flipping it off is undoable through the existing stack.

**Findings**

- **Disabling the PC toggle silently wipes character name + traits + portrait.** `src/lib/features/ship/PlayerCharacterSection.svelte:59` and `src/lib/state/workspace.svelte.js:614,633-636` together mean a single click on the "This ship is captained by my character" checkbox can drop a paragraph of traits and a portrait. The action is undoable, but there is no confirmation, no inline warning when traits/name/portrait are populated, and the help text in the disabled state (`PlayerCharacterSection.svelte:67-70`) describes the *enabled* implications only.
  - **Recommendation:** when the toggle is being switched off and any of `characterName`, `traits`, or `portraitImageId` is non-empty, `window.confirm` ("Disabling this clears the character name, traits, and portrait. Undo will restore them. Continue?") before the commit. Cheapest version: do the confirm in the `<input onchange>` handler; if the user cancels, skip the call to `setPlayerCharacterEnabled`. Also reword the disabled-state body so it mentions undo recovers prior data.
- **Blanking the ship-name input drops the edit silently.** `src/lib/state/workspace.svelte.js:319-322` early-returns for an empty name, but `src/lib/features/ship/ProfileSection.svelte:36,69-73` still leaves the input visually empty (the writable `$derived` retains the user's typing because `ship.name` didn't change). This is a small but real "I tried to do a thing and got no feedback" path, even if the canonical state is correct.
  - See Heuristic 9 for the recommended fix.

### 4. Consistency and Standards — PARTIAL

- [x] Buttons, inputs, typography, spacing, and color usage are uniform
- [ ] Primary/secondary/destructive button styles are used consistently — see ghost-vs-danger finding
- [x] Terminology is uniform
- [x] Platform conventions respected (native `<dialog>`, `<input type=file>`, `<input type=radio>` with `<fieldset>` + `<legend>`)
- [x] Iconography style is consistent (same line-stroke and viewbox conventions as v0.1)

**Evidence**

- `src/lib/features/ship/ProfileSection.svelte:62-241,OfficerCard.svelte:108-150,PlayerCharacterSection.svelte:111-133` — every form row uses the shared `Field` + `NumberStepper` / `<select>` / `<input>` primitives; help text and labels go through `Field`. Same shape as `AddShipDialog.svelte`.
- `src/lib/features/ship/OfficerCard.svelte:121-148` — the status segmented control is a real `<fieldset>` + `<legend class="sr-only">` containing native `<input type="radio">`s, with `for`/`<label>` semantics handled by wrapping the radio inside its label. This is the platform-correct way to build a radio group and was the deliberate replacement for the custom segmented control sketched earlier in the milestone.
- `src/lib/features/ship/ShipDetail.svelte:140-153` — the "Save ship" + "Remove" pair in the detail header reuses the `SaveShipButton` primitive and a plain `Button` of `variant="ghost"`, matching v0.1 conventions for primary/passive button pairs.
- `src/lib/features/ship/ProfileSection.svelte:233-237,OfficerCard.svelte:87-91,PlayerCharacterSection.svelte:103-107` — every "remove this portrait" affordance is a `Button variant="ghost" size="sm"` next to the `ImageUpload`. Three places, identical shape.

**Findings**

- **All "Remove" affordances use `variant="ghost"`, including ship-level removal.** `src/lib/features/ship/ShipDetail.svelte:141` removes the entire ship from the workspace; `src/lib/features/ship/ProfileSection.svelte:234`, `OfficerCard.svelte:88`, and `PlayerCharacterSection.svelte:104` remove a single portrait each. Visually, all four are styled identically. The four actions are not equivalent in stakes: portrait removal is a one-click undoable swap, ship removal forces the user to re-load the file from disk. The `Button` primitive has a `danger` variant (`src/lib/ui/Button.svelte:25`) that nothing currently uses.
  - **Recommendation:** switch the ship-level Remove in `ShipDetail.svelte:141` to `variant="danger"` and rename the label "Remove from workspace" so it reads differently from portrait Remove. Leaves portrait Removes as ghost (their stakes match ghost weight) and finally lights up the `danger` variant for its intended use.

### 5. Error Prevention — PARTIAL

- [x] Forms validate inputs before submit
- [x] Sensible defaults reduce required input
- [x] Inputs are constrained
- [x] Destructive actions have a confirmation step proportional to risk — except PC toggle (see H3)
- [x] State machines reject invalid transitions
- [ ] Free-text inputs are bounded — no max length on Ship name / Type / Character name / traits

**Evidence**

- `src/lib/state/workspace.svelte.js:408-413` — `setShipHpMax` floors the input at 1 and atomically clamps `hp.current` so it can never exceed the new max; `setShipHpCurrent` (lines 431-433) clamps to `[0, hp.max]`. A user dragging the max stepper down past current HP can't end up with `current > max`.
- `src/lib/features/ship/ProfileSection.svelte:131-156` — the same constraints surface in the UI: current is clamped via `max={ship.hp.max}` on the stepper, max can't drop below 1.
- `src/lib/features/ship/ProfileSection.svelte:159-167` — Explosion DC clamped to 1–40; speed knots/squares 0–99 (lines 105-126); weapon counts 0–99 (lines 173-184); rank 1–5 in `OfficerCard.svelte:110-117`.
- `src/lib/features/ship/OfficerCard.svelte:121-148` — using a true `<input type="radio">` group means only one status can be selected, even if the surrounding code drifts.
- `src/lib/state/workspace.svelte.js:646-660` — `setPlayerCharacterFields` substitutes `ship.name` for an emptied character name (line 653), so the PC name never persists as the literal empty string.
- `src/lib/state/workspace.svelte.js:319-321` — `setShipName` rejects an empty/whitespace name as a no-op, preventing a nameless ship from being committed.

**Findings**

- **Ship name accepts arbitrarily long input.** `src/lib/features/ship/ProfileSection.svelte:65-73` has no `maxlength` on the input (and the underlying mutator only trims, not truncates). The ship name becomes the file stem on save (per `README.md:30`), so a 500-character name produces an unwieldy filename and a hard-to-read rail card. Same for Type, Character name, and Traits, though those have less downstream impact.
  - **Recommendation:** add `maxlength="80"` to the ship-name and character-name inputs and `maxlength="40"` to the type input. `Traits` can stay unbounded (it's a textarea and is intentionally a notes field) but consider a soft "X / 2000" counter shown via help text once the user crosses ~1500 chars. Filename truncation in the persistence layer is an additional belt-and-braces fix but isn't covered here.
- **Officer Rank stepper allows manual typing past the cap.** `src/lib/ui/NumberStepper.svelte:26-34` only re-clamps in `oninput`, but a paste of "9" into a `min=1, max=5` field will be clamped to 5 — confirmed correct on read. Mutator (`workspace.svelte.js` `setOfficer`) accepts whatever the input emits but the call site (`OfficerCard.svelte:116`) typecasts via `/** @type {any} */` without re-clamping; the stepper's existing clamp is what protects this. Documenting as a defensive note: if the stepper ever changes shape, rank validation needs to land in the mutator.

### 9. Help Users Recognize, Diagnose, and Recover from Errors — PARTIAL

- [x] Errors are visually distinct (color, icon, position) without relying on color alone
- [x] Error copy is in plain language
- [x] Errors say what went wrong AND what to do next
- [x] Inline errors appear next to the field at fault — except in the new inline editor (see finding)
- [x] Network/system errors offer retry; permission errors offer the path to fix — N/A

**Evidence**

- `src/lib/ui/Toast.svelte:21,33` — error toasts still use `role="alert"`, `aria-live="assertive"`, and a paired Dismiss; `ImageUpload` (still the canonical source for upload failures) routes through it for both the over-cap rejection and the read failure.
- `src/lib/ui/ImageUpload.svelte:43-67` — three distinct toasts: hard-cap rejection, soft warning, and workspace-budget warning. Each names the file and the next step.
- `src/lib/features/fleet/AddShipDialog.svelte:53-54,143-144` — the "name is required" precedent (touched-state + after-blur inline error via `Field.errorText`) is still in place and is the model the inline editor should mirror when it grows real validation.
- `src/lib/state/workspace.svelte.js:319-321,342-345,408-413,431-433,646-660` — mutators are defensively coded so an invalid value can't slip into state. This is good error *prevention*, but it has a side effect: the user gets no signal when a no-op happened.

**Findings**

- **The inline editor swallows invalid edits silently.** Two concrete cases:
  1. `src/lib/features/ship/ProfileSection.svelte:36,39-41` + `src/lib/state/workspace.svelte.js:319-321` — clearing the ship name and blurring is a no-op, but the input keeps the user's empty string until the next external change to `ship.name`. The header still shows the canonical name (correct), but the input field reads as if the empty value persisted (incorrect impression).
  2. `src/lib/features/ship/PlayerCharacterSection.svelte:32-36` + `src/lib/state/workspace.svelte.js:646-660` — clearing character name and blurring substitutes `ship.name` server-side (good), but the input then doesn't visibly re-sync because the writable `$derived` still holds the user's empty string.
  - **Recommendation:** in `ProfileSection` and `PlayerCharacterSection`, on blur after a no-op or substituted commit, re-sync the local draft to the canonical value. Cheapest implementation: in the blur handler, after the mutator call, reset `nameDraft = ship.name` (and equivalent for traits / character name). Optional second pass: surface a one-shot inline `Field.errorText` ("Ship name can't be empty.") for the empty-name case, mirroring the AddShipDialog's `nameTouched` pattern, so the user gets an explicit signal instead of an automatic restore.
- **Autosave error state still has no retry affordance.** `src/lib/features/workspace/AutosaveIndicator.svelte:9-35` reports the error in label + dot but offers no way to nudge a re-flush. New in v0.2, the inline editor produces autosave traffic on every blur/change, so the indicator is more visible than it was; a stuck red dot with no exit is not great. This is shared with v0.1 — flagging as a v0.3 candidate.
  - **Recommendation:** when `state === 'error'`, render a small retry button next to the indicator that calls `autosaver.flushNow()`. Defer to v0.3.
