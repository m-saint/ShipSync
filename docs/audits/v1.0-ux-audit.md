# UX Heuristic Audit — ShipSync v1.0 Readiness + Preferences

**Scope:** the v1.0 delta on top of [`v0.9-ux-audit.md`](./v0.9-ux-audit.md). v1.0 is a combination of "Slate δ — readiness" (accessibility plumbing, dialog hardening, workspace safety guards) and "Slate ε — preferences" (filter-inversion dark mode, density toggle, autosave cadence, carry-forward defaults), plus a new `.shipsync.bundle.json` archive format for whole-workspace persistence. Findings already addressed at v0.1–v0.9 are referenced rather than re-litigated.
**Date:** 2026-04-27
**Surfaces inventoried:** 1 new dialog + 1 new confirm dialog + 1 new toolbar button + 2 new state modules + 2 new persistence modules + 1 focus-trap utility + cross-cutting accessibility plumbing in Dialog / Toast / tokens.css

Components inventoried:

- `src/lib/ui/Dialog.svelte` — centralized Escape-to-close, focus-trap wiring, window-level keydown backup
- `src/lib/ui/focus-trap.js` — new keyboard focus trap utility (DOM-ordered, jsdom-tolerant)
- `src/lib/ui/ConfirmDialog.svelte` — new reusable two-button confirm modal (ghost + danger by default)
- `src/lib/ui/Toast.svelte` — outer container always rendered so the `aria-live` region is in the DOM before the first toast
- `src/lib/ui/tokens.css` — `prefers-reduced-motion` global, `data-color-scheme="dark"` filter inversion, `data-density="compact"` rules, Tailwind v4 `@custom-variant` hooks
- `src/lib/state/settings.svelte.js` — new reactive store (`colorScheme`, `density`, `autosaveDebounceMs`, `carryOverOnCharter`) persisted to `localStorage` at `shipsync.settings.v1`
- `src/lib/features/workspace/SettingsDialog.svelte` — new gear-icon dialog surfacing all four settings
- `src/lib/features/workspace/WorkspaceTopBar.svelte` — adds the `Preferences` button and the `Save bundle` button
- `src/lib/features/fleet/SaveBundleButton.svelte` — new toolbar button that exports the workspace as `.shipsync.bundle.json`
- `src/lib/features/workspace/LoadShipsButton.svelte` — extended to sniff bundle vs ship payloads and gate bundle loads behind a `ConfirmDialog`
- `src/lib/features/fleet/AddShipDialog.svelte` — adds the donor-driven carry-forward affordance
- `src/lib/features/ship/ShipDetail.svelte` — replaces the `window.confirm` for ship removal with `ConfirmDialog`, with dirty-aware copy
- `src/lib/features/activityLog/ActivityLog.svelte` — surfaces the pruned-action count when the undo cap clips older history
- `src/App.svelte` — applies `data-color-scheme` and `data-density` to `<html>` based on settings (and OS `prefers-color-scheme` when in `auto`)

Plumbing in scope (state, persistence, derivations — not UI surfaces):

- `src/lib/persistence/bundleFile.js` — new `.shipsync.bundle.json` schema (`shipsync.bundle/v1`), `makeBundleFile`, `serializeBundleFile`, `bundleFilename`, `downloadBundleFile`, `isBundlePayload`, `parseBundleFile`. The parser returns an `issues[]` array (`severity: 'error' | 'warn'`) so the load button can surface partial-load warnings without blocking
- `src/lib/persistence/autosave.js` — `makeAutosaver` now accepts `debounceMs` as `number | () => number` so the SettingsDialog choice can apply on the next mutation without recreating the autosaver
- `src/lib/state/workspace.svelte.js` — `UNDO_LIMIT` raised to 200; `stackInfo.prunedCount` + `prunedActionCount()` exported so the Activity Log can surface the "older history pruned" hint; `IMAGE_STORE_MAX_BYTES` (10 MB) + `imageStoreSizeBytes()` + `projectImageStoreOverflow()` block oversized image uploads with a warning toast; `loadBundleIntoWorkspace` replaces the workspace inside a single undoable commit and stamps `lastSavedAtByShipId` on every imported ship

## Findings by Heuristic

### 1. Visibility of System Status — PASS

- [x] Toast container's `aria-live="polite"` region now exists in the DOM unconditionally (`Toast.svelte:21`–`:27`), with a comment at `:12`–`:20` documenting why: screen readers only announce content added to a live region that was *already present* when the announcement happened, so gating the container on `ui.toasts.length > 0` would have caused the very first toast to land silently. This change closes the only remaining a11y gap from the v0.9 audit's H1 sample.
- [x] Save-bundle and load-bundle outcomes route through `pushToast` with kind-appropriate severity. The success path reports the actual filename written (`SaveBundleButton.svelte:46`–`:50`); the failure path surfaces the underlying error message (`:52`–`:56`); a partial-load surfaces both a success ("Bundle loaded (3 ships).", `LoadShipsButton.svelte:154`–`:158`) and a follow-up warning toast that lists the first three issues from the bundle parser ("Bundle had a few quirks", `:160`–`:168`) so the user knows the workspace state is real but slightly different from the file on disk.
- [x] The undo-cap boundary is observable in the Activity Log itself: when `prunedCount > 0`, the log appends a soft hint ("3 earlier actions pruned to keep history light — those steps can't be undone.") at the bottom of the visible list (`ActivityLog.svelte:240`–`:243`). This is a non-intrusive surface (italic, ink-500, below all entries) so existing-session users don't see it at all, but a long-running session never silently drops history without explanation.
- [x] Settings choices apply with no manual refresh: `colorScheme` writes through to `<html data-color-scheme>` immediately (`App.svelte:41`–`:57`), `density` to `<html data-density>` (`:59`–`:62`), and the autosave debounce is read on every `schedule()` call via the `() => settings.autosaveDebounceMs` getter wired in `Dashboard.svelte:46`. The user gets visible confirmation that the choice took (the next autosave fires at the new cadence) without any explicit "applied" toast — the system status *is* the live behavior.
- [x] Image-cap rejections fire their own toast with formatted byte counts ("ship.png is too big — workspace would hold 11.4 MB, cap is 10.0 MB. Trim or replace existing portraits/flag art first.", `workspace.svelte.js` via `pushImageCapToast`), mapping the soft cap to a concrete "what the user just tried" + "what the cap is" + "what to do next" sentence. The toast routes through the warning kind so it stays visible long enough to read.
- [x] Bundle-replace `ConfirmDialog` (`LoadShipsButton.svelte:203`–`:216`) interpolates the workspace size into the confirm-button label ("Replace fleet (3 ships)"), and the body lists both the file being loaded and what's at risk ("Loading 'shipsync-fleet--…json' will discard your current 2 ships and the active scene…"). The user reads exactly what's about to happen before clicking the destructive button.

### 2. Match Between System and the Real World — PASS

The v1.0 copy keeps the v0.1 tone split: nautical in headlines / prose, neutral in form labels and field names.

- Settings dialog title is plainspoken ("Preferences", `SettingsDialog.svelte:65`); description sets the persistence expectation in human terms ("Tunes that follow you between sessions on this device. Saved automatically.", `:66`). The autosave cadence presets use phenomenological labels — "Live (3/4s)", "Brisk (2s)", "Steady (10s)", "Relaxed (30s)" (`settings.svelte.js:60`–`:65`) — so the user reads about how the app feels, not just a number.
- Bundle filename pattern (`shipsync-fleet--<iso>.shipsync.bundle.json`, `bundleFile.js:128`) sorts naturally on disk and reads as a date the user can scan; the per-ship file format has used the same pattern since v0.1, so the captain's mental model carries forward.
- Carry-forward affordance interpolates the donor ship's name into the checkbox copy ("Carry forward Lassie's bridge crew & supplies", `AddShipDialog.svelte:310`) so the user never has to remember which ship is acting as the template — the donor is named at the moment of decision.
- Bundle confirm-dialog body uses the captain's nouns ("discard your current 2 ships and the active scene", `LoadShipsButton.svelte:207`); confirm label encodes the action *and* the count ("Replace fleet (3 ships)", `:209`–`:211`); cancel label is "Keep current" — both options read as actions, not modal verbs.
- Pruned-history hint uses plural-aware copy and frames the outcome ("3 earlier actions pruned to keep history light — those steps can't be undone.", `ActivityLog.svelte:241`–`:243`), so the user understands the cap as a tradeoff rather than an error.

### 3. User Control and Freedom — PASS

- [x] Centralized Escape: every modal in the workspace inherits the same Escape-to-close path through `Dialog.svelte:38`–`:48` — a window-level keydown listener (`capture: true`) that runs in addition to the native `<dialog>` `cancel` event. The defensive backup catches the case where automation harnesses or custom input handlers swallow the cancel event (Svelte 5 input components occasionally do); without it, the user could end up trapped in a modal with no exit. The trap-and-listener teardown is colocated with the `$effect` so unmount can't leak.
- [x] Focus stays inside open modals via `trapFocus(dialogEl)` (`Dialog.svelte:37`, `focus-trap.js:23`–`:57`). The trap re-queries focusable descendants on every Tab so dynamically-rendered controls (e.g. the two-click Confirm/Cancel swap in ShoreLeaveDialog, the Restore-defaults state in SettingsDialog) keep cycling through the currently-rendered set. `collectFocusables` explicitly sorts results by `compareDocumentPosition` (`focus-trap.js:103`–`:108`) because some implementations (notably jsdom) return matches grouped by the comma-separated selector position rather than DOM order.
- [x] `prefers-reduced-motion` is honored globally in `tokens.css:135`–`:144`. Animations, transitions, and scroll-behavior are collapsed to `0.01ms` (rather than `0`) so any UI that depends on the final transition state — chevron rotations on disclosures, segmented-control highlight slides — still snaps to the right end-state. Users with motion sensitivity get the full app without per-component plumbing.
- [x] Destructive ship-remove now goes through `ConfirmDialog` (`ShipDetail.svelte:328`–`:338`) instead of the legacy `window.confirm` path. The body interpolates differently when the ship is dirty ("Lassie has unsaved edits since the last file save…", `:332`) versus clean ("Lassie will leave the workspace…", `:333`), and the confirm label changes label too ("Discard changes & remove" vs "Remove"). The dirty-aware path explicitly tells the user "This isn't covered by Undo." so the user can save first and then come back.
- [x] Bundle replace is gated on a `ConfirmDialog` whenever the workspace already has ships (`LoadShipsButton.svelte:97`–`:105`). When the workspace is empty, the load applies silently — no false friction on the empty-workspace case. The replace ITSELF is undoable via `loadBundleIntoWorkspace`'s single `commit()` (`workspace.svelte.js`), so even a confirmed replace can be reversed with Cmd+Z.
- [x] Settings dialog has a `Restore defaults` button in the footer (`SettingsDialog.svelte:159`) so the user can always get back to the v1.0 baseline (auto color scheme, comfortable density, 750ms autosave) without remembering what each individual default was. `resetSettings()` (`settings.svelte.js:200`–`:203`) also persists the reset so the choice survives a page reload.

### 4. Consistency and Standards — PASS

- [x] `ConfirmDialog` is reused for both destructive paths that surfaced in v1.0 (ShipDetail's remove, LoadShipsButton's bundle-replace). Both invocations use `confirmVariant="danger"` (`ShipDetail.svelte:336`, `LoadShipsButton.svelte:213`) so the danger affordance reads the same color and weight in both places. The component default is `confirmVariant="danger"` because every existing caller is asking before discarding state (`ConfirmDialog.svelte:28`); future non-destructive confirms can opt out with `"primary"` or `"ghost"`.
- [x] Settings segmented controls render with the same Tailwind shape (`inline-flex rounded-md border border-surface-300 bg-surface-50 p-0.5`) as the existing Activity Log category filter (`ActivityLog.svelte:153`–`:154`) and the autosave-cadence row inside the dialog itself (`SettingsDialog.svelte:73`, `:96`, `:122`). All three follow the same selected-state pattern (`bg-ink-900 text-ink-50` for the active option) so a user who learns the pattern in one place reads it the same in another.
- [x] Bundle and ship file naming follow the same stem pattern (`bundleFilename` at `bundleFile.js:125`–`:130` mirrors the per-ship filename rule). Both produce ISO-stamped slug-cased names with the `.shipsync.json` / `.shipsync.bundle.json` double-extension; both use the same `sanitizeStem` shape. A captain who's used "Save all" will recognize the pattern immediately when they save their first bundle.
- [x] The Preferences button (`WorkspaceTopBar.svelte:84`–`:94`) uses `variant="ghost"` to match the other tertiary toolbar actions (Shore leave, Print fleet) and a conventional gear icon. Its `aria-label="Preferences"` + `title` tooltip pair follow the same accessibility shape every other icon-bearing button in the toolbar uses.
- [x] Bundle parser issues use the same `{ severity: 'error' | 'warn', message: string }` shape (`bundleFile.js:40`–`:45`) that the per-ship `parseShipFiles` already emits, so the LoadShipsButton can route both to the same toast renderer (`LoadShipsButton.svelte:88`–`:93` for bundles, `:131`–`:140` for ships) without a special case.

### 5. Error Prevention — PASS

- [x] Bundle parser refuses partial inputs. JSON parse failure → error issue + `ok: false` at `bundleFile.js:194`–`:209`. Missing schema marker → error at `:211`–`:223`. Missing `ships` map → error at `:241`–`:248`. Future schema versions surface a warn ("Bundle was written by a different schema (shipsync.bundle/v2); loading anyway.", `:228`–`:233`) rather than refusing — forward-compat is intentional, but the user is told.
- [x] Image-store cap projection runs *before* commit. `projectImageStoreOverflow` (`workspace.svelte.js:691`–`:699`) computes the would-be byte total accounting for the bytes that pruning will free (the replaced image's bytes don't count toward the projection because pruning will drop them right after the commit lands), so legitimate replacement uploads — the common case — don't get rejected just because the new and old image temporarily co-exist. The check fires from every image setter (`setShipPortrait` at `:998`, `setOfficerPortrait` at `:1114`, `setPlayerCharacterPortrait` at `:1219`, `setShipFlagArt` at `:2433`) so no upload path can sneak past.
- [x] Undo-stack cap (`UNDO_LIMIT = 200` at `workspace.svelte.js:53`) is enforced in the same loop that pushes new entries (`:204`–`:207`). The ceiling protects long sessions from unbounded memory + autosave growth; older entries are dropped FIFO so the most recent 200 actions are always undoable. `prunedCount` (`stackInfo.prunedCount`) accumulates across the session and surfaces in the Activity Log as the "older history pruned" hint at `ActivityLog.svelte:240`–`:243`.
- [x] Settings setters validate inputs. `setColorScheme` rejects unknown values silently at `settings.svelte.js:160`; `setDensity` rejects unknown values at `:168`. `setAutosaveDebounceMs` clamps into `[MIN_AUTOSAVE_DEBOUNCE_MS, MAX_AUTOSAVE_DEBOUNCE_MS]` at `:182` (so a hand-edited localStorage value of `0` becomes `250` rather than DDoSing localStorage with per-keystroke writes) and silently drops non-finite numbers at `:181`. The `normalizeSettings` boot path (`:92`–`:116`) re-runs the same validators against the parsed payload so a corrupted localStorage entry can't crash the app on first load.
- [x] Bundle replace requires a confirm. `LoadShipsButton.svelte:97`–`:105` only auto-applies when `workspace.shipOrder.length === 0` — every populated workspace gets a `ConfirmDialog`. The dialog body (`:206`–`:208`) explicitly calls out the unsaved-changes risk ("your unsaved per-ship changes won't be written to disk first"). This is the single most destructive action in the v1.0 surface and gets the heaviest gate.
- [x] Carry-forward defaults to the captain's currently-focused ship and falls back to the most recently chartered one (`AddShipDialog.svelte:45`–`:51`), so the donor reference can never be stale or null without the affordance also hiding. The carry-forward checkbox itself only renders when both `settings.carryOverOnCharter` is on AND a donor exists (`canShowCarryOver` at `:53`), so the user never sees a checkbox they can't act on.

### 6. Recognition Rather Than Recall — PASS

- [x] Settings options carry secondary labels that obviate any "what does Auto mean?" lookup. `Auto / Follow OS`, `Light / Always`, `Dark / Always` (`SettingsDialog.svelte:50`–`:54`); `Comfortable / Default spacing`, `Compact / Tighter cards` (`:57`–`:60`). The labels are typeset two-line in the segmented control so the user never has to hover for a tooltip to understand what each option does.
- [x] Every settings field has a `helpText` that names the consequence of the choice. Dark-mode field calls out the known limitation ("Dark mode lightens portraits via a counter-invert pass; small artifacts on heavy art are expected.", `SettingsDialog.svelte:71`); density helpText explains both what compact does and when to use it ("Compact tightens ship and officer cards by ~25%. Useful on smaller screens or once a fleet exceeds half a dozen ships.", `:94`); autosave field explains the storage tradeoff ("A longer cadence reduces localStorage churn; a shorter one minimizes data loss on crash.", `:119`); carry-forward field explains both what gets copied AND that the user can opt out per-charter ("…copy the focused ship's bridge crew (officer names, ranks, statuses, notes) and current supplies onto the new ship. The Charter dialog still lets you opt out per-charter.", `:144`).
- [x] Carry-forward checkbox shows the donor ship's name at the moment of charter (`AddShipDialog.svelte:310`) so the user doesn't need to remember which ship is acting as the template. The hint underneath enumerates exactly which fields will copy ("Copies officer names, ranks, statuses, notes, portraits, and current Grub/Grog/Gear onto the new ship.", `:312`) so the user can audit the impact before clicking Charter.
- [x] Bundle confirm shows ship counts for both the current workspace and the incoming bundle ("…will discard your current 2 ships and the active scene… Replace fleet (3 ships)", `LoadShipsButton.svelte:206`–`:211`). The user reads what they're about to lose AND what they're getting in the same dialog without flipping back to the file picker.
- [x] Pruned-history hint surfaces the count ("3 earlier actions pruned…", `ActivityLog.svelte:241`–`:243`) so the user learns *why* their oldest action just stopped being undoable. They don't have to guess whether the limit is a bug, a missing feature, or a deliberate cap.

### 7. Flexibility and Efficiency of Use — PASS (was PARTIAL pre-v1.0.1)

The keyboard-shortcut surface from v0.5–v0.9.1 carried forward unchanged (Cmd/Ctrl+S for save focused ship, Cmd/Ctrl+Shift+S for save all, Cmd/Ctrl+Z / Cmd/Ctrl+Shift+Z for undo/redo, Cmd/Ctrl+Shift+P for fleet print, Cmd/Ctrl+Shift+L for shore leave, Alt+Up / Alt+Down on the focused ship rail card, see `Dashboard.svelte:103`–`:175`). v1.0 added a new toolbar action (`Save bundle`) and a new dialog (`Preferences`) that initially didn't have shortcuts; the v1.0.1 polish bundle closes both gaps.

- [x] Settings choices act as power-user customization themselves — `density: compact` is the equivalent of "tightens the list view" in many apps, and `autosaveDebounceMs` lets a user with a slow disk relax the cadence without recompiling.
- [x] Carry-forward defaults short-circuit a common chore: chartering the third ship in a campaign no longer means re-typing the same officer roster.
- [x] The Bundle save/load round-trip is a flexibility win — one captain can hand a fully-loaded session over to another in a single file rather than N ship files plus a manual scene-recreation step.
- [x] **Closed in v1.0.1 (F7.1):** `Cmd/Ctrl+Shift+B` is now wired to `Save bundle` in `Dashboard.svelte`'s keydown effect, with the same shape as `Cmd+Shift+S` / `Cmd+Shift+P` / `Cmd+Shift+L` — empty-workspace toast ("No ships aboard. Load at least one ship before saving a bundle."), gating on `shipOrder.length`, and the bundle write inlined so the shortcut works even when the toolbar button is offscreen. The `SaveBundleButton` tooltip now carries the shortcut glyph via `shortcutHint('B', { shift: true })`. README keyboard shortcuts table updated.
- [x] **Closed in v1.0.1 (F7.2):** `Cmd/Ctrl+,` (no Shift; the conventional macOS / VSCode shortcut) is now wired in `Dashboard.svelte`'s keydown effect to `openDialog('settings')`. The `WorkspaceTopBar` Preferences button tooltip now carries the shortcut glyph via `shortcutHint(',')`. No toast (the dialog is its own confirmation) and no preconditions (preferences are workspace-independent). README keyboard shortcuts table updated.

### 8. Aesthetic and Minimalist Design — PASS

- [x] SettingsDialog stays sparse: four labeled fields, one footer with `Restore defaults` left-aligned and `Done` right-aligned (`SettingsDialog.svelte:158`–`:162`). No tabs, no sub-pages, no "Advanced" disclosure — the v1.0 settings surface is small enough to live as one screen, and putting it on one screen avoids the "where did I see that toggle?" recall cost.
- [x] No competing CTAs on the new toolbar surface. Shore leave / Save all / Save bundle / Print fleet / Open file are all `variant="ghost"` or `"secondary"`; only `Add ship` keeps the `variant="primary"` slot (`WorkspaceTopBar.svelte:74`). The eye lands on the one charter affordance even after the bar grew an extra button.
- [x] Bundle confirm dialog is the v1.0-fitting size (`size="sm"` at `ConfirmDialog.svelte:48`) rather than the default `md`. Confirm prompts shouldn't drown the user in chrome — the smaller surface lets the body text fill the dialog and keeps the action buttons close to the eye-line.
- [x] The pruned-history hint is small enough to stay out of the way (`text-[11px] text-ink-500 text-center border-t border-surface-200/70 italic`, `ActivityLog.svelte:241`) but distinct enough to be findable when the user goes looking. It only renders when `prunedCount > 0`, so existing users with shorter sessions never see it at all.
- [x] Carry-forward checkbox lives behind the `settings.carryOverOnCharter` flag (`AddShipDialog.svelte:301`), so users who never opt in never see the affordance. The dialog stays four fields long for the basic case; advanced charterers with the setting on get one extra row.

### 9. Help Users Recognize, Diagnose, and Recover from Errors — PASS

- [x] Image-cap toast names what the user did, what's blocked, and what to do next ("ship.png is too big — workspace would hold 11.4 MB, cap is 10.0 MB. Trim or replace existing portraits/flag art first."). Three sentences, three signals: action, constraint, recovery.
- [x] Bundle parse errors return a structured issue list with severities (`bundleFile.js:40`–`:45`). The first error becomes the toast title body ("Could not load shipsync-fleet--…json: This file does not look like a ShipSync bundle. Expected `schema: \"shipsync.bundle/v1\"`."), so the user reads the actual mismatch rather than a generic "load failed". Warnings are surfaced separately ("Bundle had a few quirks") so a partial-load case doesn't read as a hard failure.
- [x] Bundle parse handles the most common mistakes gracefully. Stale ship ids in `shipOrder` are dropped with a warn ("Dropped 2 stale ship ids from shipOrder.", `bundleFile.js:262`–`:267`); ships missing from `shipOrder` are appended with a warn ("Ship 'foo' wasn't listed in shipOrder; appended at the end.", `:271`–`:275`); a missing scene defaults to idle with a warn ("No scene found in bundle; defaulted to an idle scene.", `:282`–`:287`). Each of these is recoverable in-place — the user gets the data they asked for plus a heads-up about the irregularity.
- [x] Settings load is robust against corruption (`settings.svelte.js:124`–`:133`): invalid JSON, missing fields, and out-of-range values all fall through to defaults so a hand-edited localStorage entry can't crash the app on boot. The user gets the v1.0 baseline without any explicit error message — a consistent recovery rather than a noisy one.
- [x] Settings setters validate inputs and silently no-op on bad input (`setColorScheme`, `setDensity`, `setAutosaveDebounceMs` at `settings.svelte.js:159`–`:186`). A future caller passing the wrong shape doesn't poison the persisted store; the value just doesn't take.

### 10. Help and Documentation — PASS

- [x] Every settings field has explicit `helpText` rendered as a `Field` description (`SettingsDialog.svelte:71`, `:94`, `:119`, `:144`). The dark-mode helpText specifically calls out the known filter-inversion limitation ("…small artifacts on heavy art are expected") so users with image-heavy workspaces aren't surprised when their flag art looks slightly off.
- [x] Bundle button tooltip names both the format and what's in it (`SaveBundleButton.svelte:30`–`:35` builds the title), reading "Save the entire workspace (3 ships + scene + images) as a .shipsync.bundle.json archive." The user understands the file contents before clicking — no after-the-fact "wait, what's in this?" moment.
- [x] Preferences button tooltip enumerates the controls inside ("Preferences (color scheme, density, autosave cadence)", `WorkspaceTopBar.svelte:87`) so a user encountering the gear icon for the first time knows what's behind it.
- [x] LoadShipsButton tooltip explicitly mentions both file formats it accepts ("Load .shipsync.json ships or a .shipsync.bundle.json fleet archive", `LoadShipsButton.svelte:187`), so the captain doesn't need to remember which file pattern goes with which loader.
- [x] Carry-forward checkbox has a sub-label that enumerates copied fields explicitly ("Copies officer names, ranks, statuses, notes, portraits, and current Grub/Grog/Gear onto the new ship.", `AddShipDialog.svelte:312`). The user audits the action before committing.
- [x] README is the canonical reference for keyboard shortcuts, file formats, and the v1.0 status block (updated as part of this milestone).
- [x] **Closed in v1.0.1 (F10.1):** The SettingsDialog `Carry forward on Charter` helpText at `SettingsDialog.svelte:144` now reads "officer names, ranks, statuses, notes, **portraits**" — matching the AddShipDialog sub-label at `:312`. Both surfaces now describe the same set of fields, removing the small surprise a user reading only the Preferences panel could otherwise hit when their flagship's portrait clones onto the new ship.

## v1.0.3 — Post-Deploy Usability Pass

After the first GitHub Pages deployment a real-world usability session surfaced nine UX defects that the v1.0 audit didn't catch (the audit had been run against the dev surface where the regressions hadn't manifested yet — a few of these only show up at production-style viewport widths or with a ship loaded). All nine are addressed in this pass.

### Findings (code-level)

- **B1: Full dark palette rewrite.** `tokens.css` no longer applies `filter: invert(1) hue-rotate(180deg)` to `<html>` under `data-color-scheme='dark'`. Instead, the dark scheme overrides the palette tokens directly — `--color-surface-50…900`, `--color-ink-50…900`, `--color-brass-*`, `--color-crimson-*`, `--color-amber-*`, plus a hand-tuned `dialog::backdrop` rule. Dialogs, popovers, segmented controls, and image art all follow the theme without inversion artifacts. `SettingsDialog.svelte` helpText updated to drop the inversion caveat. `App.svelte` and `settings.svelte.js` JSDoc clarify that print views still drop the dark attribute (cream-on-ink is wasteful on paper).
- **B2: Lock body / lock html, give the rails internal scroll.** The dashboard root in `Dashboard.svelte` is `h-dvh overflow-hidden`; FleetRail (`overflow-y-auto`), ShipDetail (`overflow-y-auto overflow-x-hidden`), and Scene rail (`overflow-y-auto`) each scroll their own contents. The document itself never grows past one viewport, so cursor position no longer leaks scroll to the body.
- **B3: Portal-positioned tooltip with viewport clamping.** `Tooltip.svelte` rewritten end-to-end. The bubble is `position: fixed` with a `requestAnimationFrame`-deferred `reposition()` that auto-flips top↔bottom or left↔right when the requested side would clip, then clamps the primary axis to `[8px, viewport - bubble - 8px]`. `max-width: min(calc(100vw - 32px), 320px)` caps the bubble at half-screen so a long-running rule hint never exceeds half the screen on a phone-sized viewport. Scroll/resize listeners hide the bubble immediately so a stale-rect bubble can't float away from its anchor. The wrapper is a `<svelte:element>` (span by default, div when `display="block"`) that forwards hover/focus/blur — the existing `<RuleTooltip>` API stayed exactly the same so no caller had to change.
- **B4: Weapon slots stack label-on-top + drop to 2 columns.** `ShipDetail.svelte`'s "Weapon slots" cluster used to render four `Field` components in a horizontal row at narrow ShipDetail widths; the labels and number steppers competed for the same 80–90px column and ended up unusable. The cluster is now a 2×2 grid with each `Field` rendering its label on top of the stepper (the layout the rest of the form already uses), so each slot has its own ~180px column and the number stepper rendering is comfortable.
- **B5: Officer status as `<button aria-pressed>` instead of `<input type="radio">` + `<label>`.** `OfficerCard.svelte`'s status segmented control is now three `<button type="button" aria-pressed={active}>`s with no associated radio inputs. The previous radio-and-label approach had a hidden `<input>` that the browser would auto-scroll into view on `:checked` change; on a long ship form, that auto-scroll was the source of the unexpected jump. The button approach has no hidden focus target and the surface stays put when the user picks Stricken or Dead.
- **B6: Status fills the form column on narrow cards.** Same control as B5; the layout fix is that within `OfficerCard.svelte` the form column is a vertical stack (Name → Rank → Status → Duties), not a Rank-beside-Status row. With `flex w-full` and three `flex-1` buttons inside an `overflow-hidden rounded-md border` group, all three options divide the form column equally and never overhang the card. Verified at viewport width 1147px (the production-test width that originally surfaced the bug): three status buttons fit inside every officer card, captain or otherwise, with no clipping.
- **B7: ShipDetail explicit `overflow-x-hidden`.** The center pane's `<section aria-label="Ship detail">` carried `overflow-y-auto` only; a long input value or wide control could let the section grow past its column and leak a 1–2px horizontal scroll. Adding `overflow-x-hidden` to the existing `flex-1 min-w-[22rem] overflow-y-auto` line locks the pane to vertical scrolling only.
- **B8: Size autofills from the canonical type profile.** `AddShipDialog.svelte` adds a `sizeIsCustom` flag (default false). The `$effect` that already auto-filled mobility/HP/crew/explosion DC from a recognized type now also writes `form.size` from the type profile when `sizeIsCustom` is false, and flips `sizeIsCustom = true` if the user hand-edits Size. The `<datalist>` for the type field was also corrected to use canonical names (Sloop, Frigate, Galleon, …) so the in-input suggestions match the keys the profile lookup uses.
- **B9: Body + html `overflow: hidden` while any dialog is open.** `tokens.css` now carries `html:has(dialog[open]), html:has(dialog[open]) body { overflow: hidden; }`. The selector matches any open native `<dialog>` (the platform manages the `open` attribute on `showModal()` / `close()`), so every modal in the app — Charter, Settings, Confirm, Shore leave, Apply flag — locks the page underneath it without any per-component plumbing. Belt-and-suspender (both html and body) because Chrome/Edge attribute the document scroll to `<html>` on some viewports and Firefox/Safari to `<body>`. Also clears unintended document growth from layout-level overflow and keeps the dialog visually anchored to the surface it covered.

### Heuristic deltas

- **H1 (Visibility of System Status)**: B5's removal of the auto-scroll-on-radio-change closes a small but real "what just moved?" surprise — the user now sees their click land where they clicked, not 200px below.
- **H4 (Consistency and Standards)**: B1's palette rewrite brings the dialog/popover surfaces into the same theming pipeline as the dashboard, so the user experiences one coherent dark palette rather than "dark dashboard + light dialogs".
- **H5 (Error Prevention)**: B9 prevents a class of "I can't find the dialog" confusion where the page scrolled the dialog out of view. B8 prevents the "I have a Galleon-named Tiny ship" data-entry mistake.
- **H8 (Aesthetic and Minimalist Design)**: B3, B4, B6, B7 each remove a visual misalignment that was leaking polish — clipped tooltips, smushed weapon inputs, overflowing officer status, phantom horizontal scrollbar.
