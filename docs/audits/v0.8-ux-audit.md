# UX Heuristic Audit — ShipSync v0.8 Damage composer + Repair flow + Fleet print

**Scope:** the v0.8 delta on top of [`v0.7-ux-audit.md`](./v0.7-ux-audit.md). v0.8 is "Slate α — table efficiency" and ships three user-visible additions plus one architecture refactor: (a) a `DamageComposer` inline panel that lets a player apply hull / mettle / crew / fires deltas as a single atomic commit with a free-form source note that lands in the log entry; (b) a `RepairComposer` inline panel that pairs hull restoration with optional Grub / Grog / Gear costs as the same kind of one-commit event; (c) a `?print=fleet` route that hydrates from autosave and renders one `ShipPrintSheet` per loaded ship with `break-after: page`, paired with a workspace-toolbar `Print fleet` button that opens the route in a fresh tab; and (d) a refactor that extracts the per-ship print body from `PrintView` into a reusable `ShipPrintSheet` component (which also fixed the v0.7 `officerCasualtyTally` argument bug, see findings under H1). Findings already addressed at v0.1–v0.7 are referenced rather than re-litigated.
**Date:** 2026-04-27
**Surfaces inventoried:** 1 new top-level route + 2 new inline panels + 1 new toolbar button + 1 extracted print component + 3 changed feature components

Components inventoried:

- `src/lib/features/ship/DamageComposer.svelte` — new, inline panel for composing a multi-resource damage event
- `src/lib/features/ship/RepairComposer.svelte` — new, inline panel for composing a hull-repair-plus-supply-cost event
- `src/lib/features/ship/CombatReadinessSection.svelte` — adds the footer row with `[Take damage]` / `[Repair]` toggles plus the explanatory hint
- `src/lib/features/fleet/PrintFleetButton.svelte` — new workspace-toolbar button that opens `?print=fleet` in a fresh tab
- `src/lib/features/workspace/WorkspaceTopBar.svelte` — wires `PrintFleetButton` adjacent to `SaveAllShipsButton`
- `src/routes/FleetPrintView.svelte` — new, multi-ship print sheet (one `ShipPrintSheet` per loaded ship, hard page breaks)
- `src/routes/PrintView.svelte` — refactored to delegate to `ShipPrintSheet` for the single-ship case
- `src/routes/ShipPrintSheet.svelte` — new, pure rendering component for one ship's print body (shared by both routes)
- `src/App.svelte` — URL router extended: `?print=fleet` → `FleetPrintView`, `?print=<shipId>` → `PrintView`

Plumbing in scope (state, persistence, derivations — not UI surfaces):

- `src/lib/domain/derivations.js` — `composeDamageDeltas` / `composeDamageSummary` / `composeRepairDeltas` / `composeRepairSummary` pure helpers; `actionCategory` extended to bucket `ship.damage` and `ship.repair` under Combat
- `src/lib/state/workspace.svelte.js` — `applyCombatDamage(shipId, deltas, source)` and `applyRepair(shipId, hullDelta, costs, source)` mutators, both single-commit / single-log / clamp-aware / no coalescing

## Findings by Heuristic

### 1. Visibility of System Status — PASS

- [x] Both composer panels render a live preview of the exact log line that will be committed, computed from the same pure helper (`composeDamageSummary` / `composeRepairSummary`) the mutator uses to write the entry. The preview is wrapped in `aria-live="polite"` so screen readers re-announce on each delta change (`DamageComposer.svelte:163`–`:167`, `RepairComposer.svelte:171`–`:175`). Concrete example seen during smoke test: "Took damage on Lassie (Black Spear broadside): -4 hull, -2 mettle, +1 fire."
- [x] Each composer's header shows the *current* values for every resource it can touch — Hull / Mettle / Crew / Fires for damage; Hull / Grub / Grog / Gear for repair — so the player sees the baseline they're editing against without leaving the panel (`DamageComposer.svelte:99`–`:101`, `RepairComposer.svelte:103`–`:105`). On Lassie post-damage the repair panel header read "Hull 56/60 · Grub 0 · Grog 0 · Gear 0", which is exactly the state the clamp logic uses.
- [x] Apply buttons disable themselves when no deltas are entered (`hasAnyDamage` / `hasAnyChange`, `DamageComposer.svelte:42`, `RepairComposer.svelte:45`; gated at `:176` and `:184` respectively) and the preview slot announces "No damage entered yet." / "No repair entered yet." in the same `aria-live` region so the disabled state has a positive verbal explanation, not just a grayed button.
- [x] Each Apply button follows the established two-click confirm pattern from `ScenePanel`'s End-scene affordance: first click flips to `confirming = true` and re-labels the button "Confirm: apply damage" / "Confirm: apply repair" with a paired "Back" escape (`DamageComposer.svelte:170`–`:189`, `RepairComposer.svelte:178`–`:197`). If the underlying state changes such that the action becomes a no-op while the confirm is pending, an `$effect` collapses confirming back to false (`DamageComposer.svelte:78`–`:80`, `RepairComposer.svelte:82`–`:84`).
- [x] Number steppers in both composers tint by entered magnitude rather than by stored ship state — non-zero hull damage and ignited fires read crimson (`DamageComposer.svelte:112`, `:142`), non-zero mettle / crew / supply costs read amber (`:122`, `:132`, `RepairComposer.svelte:129`, `:139`, `:149`), entered hull restored reads sea (`RepairComposer.svelte:115`). This makes the composer-as-event-staging-area visually distinct from the editor-of-record steppers above it, even though both use the same `NumberStepper` component.
- [x] The `Print fleet` toolbar button reflects whether the action is meaningful: it disables itself when the workspace is empty and adopts an explanatory tooltip ("Load at least one ship to print the fleet."), switching to a count-aware tooltip when ships are loaded ("Open a print-ready sheet for each of the 2 ships aboard. One Cmd+P prints them all.") (`PrintFleetButton.svelte:21`–`:28`). Verified in smoke test: button rendered disabled at `?print=fleet` until Lassie was chartered, at which point the chip flipped enabled and the Save All counter ticked to "(1)".
- [x] `FleetPrintView` exposes its hydration state explicitly: "Loading fleet…" while the autosave read is in flight (`:101`), an amber banner if no autosave snapshot exists ("No local autosave found. Open the dashboard first, load your fleet, then print." `:35`–`:38`, rendered at `:96`–`:99`), and a separate amber banner if the autosave hydrated but contained zero ships ("No ships in the workspace yet. Load some in the dashboard, then come back to print." `:103`–`:105`). The action bar's Print button is gated on both flags (`disabled={!hydrated || ships.length === 0}` at `:87`).
- [x] `FleetPrintView` header reports the rendered count alongside the route name once hydrated ("Fleet sheets — 2 ships", `:69`–`:74`), so the user can sanity-check that the workspace they expected actually came across before sending the print job.
- [x] **Bug fixed during the v0.8 refactor:** `PrintView` had been calling `officerCasualtyTally(ship.officers)` since v0.7, which silently returned an all-zero tally because the helper expects the full ship object. The print sheet under-reported bridge losses. The extraction into `ShipPrintSheet.svelte` corrected the call site to `officerCasualtyTally(ship)` (`ShipPrintSheet.svelte:60`), and the same fix carries automatically to `FleetPrintView` since both routes share the component.

### 2. Match Between System and the Real World — PASS

The v0.8 copy keeps the v0.1 tone split: nautical in headlines and prose, neutral in form labels.

- The composer field labels read like the player would describe the event out loud — "Hull / Mettle / Crew / Fires" for damage with concrete role blurbs ("Damage to the ship.", "Captain's nerve hit.", "Casualties this hit.", "New fires igniting." `DamageComposer.svelte:105`–`:144`); "Hull restored / Grub / Grog / Gear" for repair with the same plainspoken framing (`RepairComposer.svelte:108`–`:151`). The Source field's placeholder advertises the register: "e.g. Black Spear broadside" for damage (`DamageComposer.svelte:157`), "e.g. shore party, carpenter's work" for repair (`RepairComposer.svelte:165`).
- Log line summaries match the spoken cadence and pull through the source verbatim: "Took damage on Lassie (Black Spear broadside): -4 hull, -2 mettle, +1 fire." (`composeDamageSummary` in `derivations.js`) and "Repairs on Lassie (carpenter's mate): +3 hull." Each unit has its sign attached so a quick scan tells you what direction the resource moved.
- The composer toggle buttons read "Take damage" and "Repair" — the verbs that actually get said at the table — instead of generic "Damage event" / "Apply repair" framings that would put the player one mental hop away from the action they meant to log.
- The toolbar button reads "Print fleet" (the noun the user would search for) with a tooltip that explains the multi-page payoff ("One Cmd+P prints them all." `PrintFleetButton.svelte:25`). The print-tab title reads "Fleet sheets — ShipSync" (`FleetPrintView.svelte:62`–`:64`), keeping the user oriented if they have multiple browser tabs open.
- The repair composer's hull help text reframes the cap in player language — "Up to +4 headroom." when there's room to repair, "Already at full HP." when there isn't (`RepairComposer.svelte:108`) — instead of forcing the user to do the arithmetic from `hp.max - hp.current`.

### 3. User Control and Freedom — PASS

- [x] Both composers pair the Apply trigger with a Cancel escape (`DamageComposer.svelte:172`, `RepairComposer.svelte:180`) — no modal interruption, no double-tap risk. The two-click confirm has its own Back escape so the player can step out of the confirm without losing entered values (`DamageComposer.svelte:184`, `RepairComposer.svelte:192`); the values reset only on a successful Apply or an explicit Cancel (`DamageComposer.svelte:44`–`:51`, `RepairComposer.svelte:47`–`:54`).
- [x] The whole composer event lands as one undoable commit (`workspace.svelte.js:applyCombatDamage` / `applyRepair`, single `commit(...)` call per mutator), so a one-click undo from the workspace toolbar fully reverses a six-stepper-edits-worth-of-deltas event. Smoke-test evidence: applying -4 hull / -2 mettle / +1 fire to Lassie produced exactly one `ship.damage` log line in the Activity Log; an Undo would restore all three resources in one step.
- [x] Composers are mutually exclusive — opening Repair while Damage is open closes Damage and vice versa (`CombatReadinessSection.svelte:47`–`:55`) — and clicking a trigger a second time closes its own panel. The user is never left wondering which inline panel the next click will affect.
- [x] `aria-expanded` on the trigger buttons (`CombatReadinessSection.svelte:237`, `:245`) gives assistive tech the same open/closed signal the visual variant swap (secondary↔primary/danger) gives sighted users.
- [x] `FleetPrintView` is read-only — it imports neither `commit()` nor any mutator from `workspace.svelte.js`, only the reactive `workspace` proxy and the autosave reader (`FleetPrintView.svelte:23`–`:24`). The sticky action bar exposes Close (`window.close()`, `:79`) and Print (`window.print()`, `:86`) and gets stripped from `@media print` (`:126`–`:128`) so it doesn't print on paper.
- [x] The `Print fleet` toolbar button uses `window.open(..., '_blank', 'noopener')` (`PrintFleetButton.svelte:31`–`:34`), not a navigation, so the user's dashboard tab stays intact and they can come back to it without losing scene state. Verified in smoke test: clicking the button opened a fresh tab while the original dashboard kept its workspace and active ship selection.

### 4. Consistency and Standards — PASS

The v0.8 surfaces stay consistent with the rest of the app.

- Both composers reuse the shared `<Button>`, `<NumberStepper>`, `<Field>`, and `<RuleTooltip>` primitives — no bespoke chrome — so they pick up the same focus rings, hover states, and disabled treatments as every other input in the dashboard (`DamageComposer.svelte:19`–`:22`, `RepairComposer.svelte:19`–`:22`).
- Confirm-button styling matches by-action semantics: damage ships in `variant="danger"` (crimson, `DamageComposer.svelte:174`, `:185`) the same way `removeShipFlag` does in `FlagsSection`, while repair ships in `variant="primary"` (brass, `RepairComposer.svelte:182`, `:193`) the same way `Add ship` does. The two-click pattern itself is borrowed directly from `ScenePanel`'s End-scene affordance, so a player who learned that gesture in v0.7 already knows the v0.8 one.
- `Print fleet` and `Save all` use the same toolbar button class (`SaveAllShipsButton.svelte:46`, `PrintFleetButton.svelte:34`–`:42` — both `<Button variant="secondary" size="md">` with a leading 14×14 icon), and they sit next to each other in `WorkspaceTopBar.svelte:46`–`:48`. The `Print sheet` per-ship action that v0.7 introduced still lives in the ship-header action row, so the toolbar button reads as the bulk equivalent and not a duplicate.
- `FleetPrintView` reuses the exact `ShipPrintSheet` component that `PrintView` uses (`FleetPrintView.svelte:110`, `PrintView.svelte:107`). Same typography, same KPI tiles, same heading order — a printed fleet sheet and a printed single sheet are visually identical on a page. The only delta is the `break-after: page` declaration on the wrapper (`FleetPrintView.svelte:134`–`:137`), which is print-only and visible in the screen view as the gap between adjacent ship blocks.
- Composer log entries route through the same `appendShipLog` and `actionCategory` machinery as every other ship action; `ship.damage` and `ship.repair` were added to the `'combat'` bucket in `actionCategory(kind)` (`derivations.js`), so the v0.7 Activity Log filter chips bucket them correctly without code changes in `ActivityLog.svelte`.

### 5. Error Prevention — PASS

- [x] Damage composer clamps every input at the *current* value of the resource it touches (`DamageComposer.svelte:110`, `:120`, `:130`, `max={ship.hp.current}` etc.) so a player can't enter -10 hull on a ship with 4 HP and end up with negative HP. The mutator clamps the same way at the state layer (`applyCombatDamage` clamps with `Math.min(requestedHull, ship.hp.current)` etc.), so a programmatic call from a future feature can't bypass the UI guard.
- [x] Repair composer clamps hull at headroom (`RepairComposer.svelte:113`, `max={headroom}` derived from `hp.max - hp.current` at `:35`) and supply costs at the ship's current supply level (`:127`, `:137`, `:147`, `max={ship.supplies.grub|grog|gear}`). Verified in smoke test: with Lassie at 56/60 hull and 0/0/0 supplies, the hull stepper accepted up to +4 ("Up to +4 headroom.") and all three supply steppers were keyboard-disabled at zero.
- [x] Source field is `maxlength="80"` on both composers (`DamageComposer.svelte:156`, `RepairComposer.svelte:164`) — same length cap as the ship-name and character-name inputs from v0.2, so a runaway paste can't blow out the log entry width or the autosave footprint.
- [x] Apply is gated on `hasAnyDamage` / `hasAnyChange` derived from the same summary helper that produces the log line (`composeDamageSummary` / `composeRepairSummary` returning `null` for an all-zero payload). A composer that previewed nothing literally cannot commit, so there's no "ghost commit with empty deltas" failure mode.
- [x] The two-click confirm doesn't fire the action on the first click — the first click only opens the confirm row, the second click is the commit (`startConfirm` → `handleApply`, `DamageComposer.svelte:69`–`:72` and `:53`–`:62`). A single accidental click on Apply is recoverable by clicking Back, with no state mutation in between.
- [x] The repair flow allows a "supplies-only" event (e.g. "the carpenter botched it but burned through grog anyway") because `composeRepairSummary` produces a non-null line whenever any one of hull / grub / grog / gear is non-zero. This is documented in the component header (`RepairComposer.svelte:14`–`:17`) so the player knows it's deliberate, not a UI bug.
- [x] `FleetPrintView` guards against the empty-snapshot edge case — both "no autosave found" and "autosave hydrated but no ships" surface their own amber banner instead of printing a blank page (`FleetPrintView.svelte:96`–`:105`). The Print button stays disabled for both cases (`:87`).

### 6. Recognition Rather Than Recall — PASS

- [x] Both composers show their target ship's current resource values inline (`DamageComposer.svelte:99`–`:101`, `RepairComposer.svelte:103`–`:105`) so the player doesn't have to remember "wait, what's Lassie at now" before entering the deltas.
- [x] The `[Take damage] [Repair]` toggles in the Combat Readiness footer are both visible at once (`CombatReadinessSection.svelte:233`–`:248`), with persistent labels and an explanatory hint above them ("Whole event in one commit — hits multiple resources, lands one log line." `:229`–`:231`). The user doesn't have to remember which menu the action lives in or hover-discover its name.
- [x] Each composer field has a persistent label rendered through the shared `<Field>` component (e.g. `<Field label="Hull" helpText="Damage to the ship.">`, `DamageComposer.svelte:105`); placeholders are reserved for examples on text inputs, never used as the only label.
- [x] The repair composer's headroom hint ("Up to +4 headroom." / "Already at full HP." `RepairComposer.svelte:108`) tells the player what the max actually is at the moment of the decision instead of relying on them to subtract `hp.current` from `hp.max` in their head.
- [x] The composer headers carry a "one commit" tag (`DamageComposer.svelte:91`–`:97`, `RepairComposer.svelte:95`–`:101`) backed by a `RuleTooltip` describing the design intent — the player can hover in the moment they need it, but the tag itself is enough of a recognition cue once they've used the panel once.
- [x] `FleetPrintView` keeps Lassie / Mercury / etc. in the same order as the dashboard's fleet rail (`workspace.shipOrder`, `FleetPrintView.svelte:48`–`:51`) so the printed packet's pagination matches the on-screen mental model. Verified in smoke test: Lassie printed first, Mercury second, matching the rail order.

### 7. Flexibility and Efficiency of Use — PASS (was PARTIAL; v0.8.1 closed both findings)

The composer flow is the v0.8 efficiency win — six clicks worth of stepper edits collapsed into one two-click commit with one log line. That's exactly what the slate was selected for. The two efficiency gaps called out at v0.8 (F7.1 keyboard shortcut for `Print fleet`, F7.2 composer auto-focus) were closed in the v0.8.1 polish bundle and are documented under "Resolutions in v0.8.1" below.

- [x] **The headline win:** in the smoke test, applying -4 hull / -2 mettle / +1 fire to Lassie went from "increment Mettle stepper twice, increment Fires stepper once, decrement Hull stepper four times" (~7 clicks across three sections, three separate log lines) to "open Damage panel, type 4/2/1, type source, click Apply, click Confirm" (7 clicks but consolidated, with a previewed log line and one undo step). The win is concentrated on the cognitive side: the player commits the *event*, not the seven *deltas*.
- [x] The single Cmd+P print path for the whole fleet is genuinely batch-action plumbing: one click on `Print fleet` opens a tab pre-populated with N ship sheets, hard page breaks between them, and a single browser print dialog covers the lot (`FleetPrintView.svelte:107`–`:113`, with `break-after: page` at `:134`–`:137`). At the table this is the difference between "I'll print everyone's sheets at the start of the session" being five tabs and five Cmd+Ps versus one tab and one Cmd+P.
- [x] `Print fleet` honors a clickable button + tooltip parity with `Save All`: both expose enabled/disabled states from workspace counts, both ship a `title=` tooltip with a one-line summary that updates with the current count, and **both now surface a keyboard-shortcut glyph** in their tooltip (`PrintFleetButton.svelte:24`–`:28`, mirroring `SaveAllShipsButton.svelte:35`–`:43`).
- [x] **F7.1 (closed in v0.8.1):** `Cmd+Shift+P` (Apple) / `Ctrl+Shift+P` (other OSes) opens the fleet print sheet in a new tab via the same `window.open(..., '_blank', 'noopener')` path the toolbar button uses (`Dashboard.svelte:127`–`:142`). The handler gates on `workspace.shipOrder.length === 0` and pushes an info toast ("Nothing to print — Load at least one ship before printing the fleet.") when the workspace is empty, mirroring the `Cmd+Shift+S` empty-workspace pattern from v0.4.1. The shortcut glyph is exposed in the `PrintFleetButton` tooltip via `shortcutHint('P', { shift: true })` (`PrintFleetButton.svelte:18`), and the README's keyboard-shortcut table is extended with the new row. Plain `Cmd+P` is intentionally **not** intercepted — the player still gets native print on whatever document is in focus; the Shift modifier is the explicit opt-in to the fleet path.
- [x] **F7.2 (closed in v0.8.1):** opening either composer (Damage or Repair) immediately focuses *and selects* the first numeric stepper input (`NumberStepper.svelte:31`–`:40`, gated by an opt-in `autofocus` prop wired from `DamageComposer.svelte` Hull stepper and `RepairComposer.svelte` Hull-restored stepper). The player can type a number directly without first tabbing past the trigger button. The `.select()` call makes type-over feel native: a placeholder `0` is highlighted, the first keystroke replaces it. Verified in smoke test: tapping `Take damage` on Lassie put the cursor in the Hull field with the `0` selected, ready for a digit.

### 8. Aesthetic and Minimalist Design — PASS

- [x] Both composer panels use a tinted card background that matches their semantic role — crimson for damage (`bg-crimson-50/40`, `DamageComposer.svelte:86`), sea (cool blue) for repair (`bg-sea-50/40`, `RepairComposer.svelte:90`) — and stay scoped to the Combat Readiness section's footer so they never crowd the rest of the ship detail.
- [x] Only one composer can be open at a time (`CombatReadinessSection.svelte:47`–`:55`), so the section never grows two panels in parallel and the page never has two competing primary-by-color CTAs (one crimson "Apply damage", one brass "Apply repair").
- [x] The composer headers run "label + one-commit tag + current-state line" and nothing else (`DamageComposer.svelte:87`–`:102`, `RepairComposer.svelte:91`–`:106`). No marketing copy, no tutorial text, no decorative iconography.
- [x] The action bar in `FleetPrintView` is exactly `Close` + `Print` (`:76`–`:91`) — same minimalist treatment as `PrintView` for the single-ship case (`PrintView.svelte:76`–`:91`). No fleet-specific extra controls, no "Print options" submenu, no nested settings.
- [x] The print sheet itself — now in `ShipPrintSheet.svelte` — uses generous typography and one-column flow with ink-300 borders for KPI tiles (`ShipPrintSheet.svelte:131`–`:158`); the same restraint v0.7 audited as PASS, now applied across N ships in the fleet view.

### 9. Help Users Recognize, Diagnose, and Recover from Errors — PASS

- [x] Apply being disabled when there's nothing to commit isn't silent — the preview slot says "No damage entered yet." / "No repair entered yet." in the same `aria-live` region (`DamageComposer.svelte:166`, `RepairComposer.svelte:174`), so a player who clicked the trigger and then forgot what to type gets a positive nudge.
- [x] Clamping is *visible*: the `+` button on a stepper grays out when the entered value hits the resource's current cap (the `NumberStepper` disables the increase button at `max`), so a player can't poke +5 hull damage on a 4-HP ship and wonder why nothing happened. The mutator-side clamp in `applyCombatDamage` / `applyRepair` is the safety net, not the user-facing affordance.
- [x] `FleetPrintView` distinguishes its three error/empty states by message rather than by lumping them together: "Loading fleet…" (in flight), "No local autosave found. Open the dashboard first, load your fleet, then print." (no snapshot), "No ships in the workspace yet. Load some in the dashboard, then come back to print." (snapshot exists but empty) — each one tells the user what's wrong AND what to do next (`:96`–`:105`).
- [x] The composer auto-recovers from a stale confirm: if the underlying state changes such that `hasAnyDamage` / `hasAnyChange` flips false while the confirm is pending (e.g. an external undo zeroed the deltas), the `$effect` reverts `confirming` to `false` so the next click can't fire a no-op commit (`DamageComposer.svelte:78`–`:80`, `RepairComposer.svelte:82`–`:84`).

### 10. Help and Documentation — PASS

- [x] Each composer ships an inline footer hint above the trigger buttons explaining the design intent: "Whole event in one commit — hits multiple resources, lands one log line." (`CombatReadinessSection.svelte:229`–`:231`). A new player who sees `Take damage` next to the existing Mettle / Crew / Fires steppers gets a one-glance answer to "wait, why is there a separate damage button?".
- [x] The composer headers carry a `RuleTooltip`-backed "one commit" tag with a longer hint text describing the clamp semantics and the source-field role (`DamageComposer.svelte:82`–`:83`, `RepairComposer.svelte:86`–`:87`). The user hovers when they want it and otherwise the tag stays a low-key textual marker.
- [x] Each numeric field has a `helpText` blurb on its `<Field>` wrapper that names the role in one phrase ("Damage to the ship.", "Captain's nerve hit.", "Casualties this hit.", "New fires igniting." for damage; "Spent on this repair." for each supply on repair). The hull-restored field's `helpText` updates between "Up to +4 headroom." and "Already at full HP." based on live state (`RepairComposer.svelte:108`).
- [x] The Source-field `helpText` is honest about its role: "Optional. Names the cause in the log entry." / "Optional. Names the repair in the log entry." (`DamageComposer.svelte:150`, `RepairComposer.svelte:158`) — the user knows up front that it isn't required and what it does.
- [x] `FleetPrintView`'s component header documents the empty-state rationale as a code comment (`:14`–`:20`), so a future contributor adding a third print route knows why the autosave-only-but-no-ships branch exists.
- [x] The `Print fleet` toolbar button has a count-aware tooltip that explains the value proposition ("Open a print-ready sheet for each of the 2 ships aboard. One Cmd+P prints them all." `PrintFleetButton.svelte:24`–`:27`).
