# UX Heuristic Audit — ShipSync v0.9 Shore Leave + Sail Order + Flag Templates

**Scope:** the v0.9 delta on top of [`v0.8-ux-audit.md`](./v0.8-ux-audit.md). v0.9 is "Slate β — fleet maneuvers" and ships three user-visible additions, all aimed at multi-ship operations the v0.5–v0.8 surfaces could only do one ship at a time: (a) a fleet-level **Shore Leave** dialog that lets the captain restore hull, top up Grub / Grog / Gear, and optionally clear scene-only conditions across selected ships in a single undoable commit; (b) **Sail Order** controls — Move-up / Move-down chevron buttons on every fleet-rail card — for reordering `workspace.shipOrder` without drag-and-drop chrome; (c) **Flag Templates** — an "Apply to other ships…" button on every flag card that copies the source flag's name / art / reputation / `isPirate` / `isFalse` onto selected target ships, with name-based dedup so the same banner can't be doubled up. Findings already addressed at v0.1–v0.8 are referenced rather than re-litigated.
**Date:** 2026-04-27
**Surfaces inventoried:** 2 new dialogs + 1 fleet-rail card refactor + 1 toolbar button + 1 flag-card action + 5 new state mutators + 1 new pure helper

Components inventoried:

- `src/lib/features/fleet/ShoreLeaveDialog.svelte` — new, fleet-level multi-ship refit composer
- `src/lib/features/fleet/ShipCard.svelte` — refactored from a single button into focus-button + sibling reorder-button column
- `src/lib/features/workspace/WorkspaceTopBar.svelte` — adds the `Shore leave` toolbar button between `AutosaveIndicator` and `SaveAllShipsButton`
- `src/lib/features/ship/ApplyFlagDialog.svelte` — new, source-pinned dialog for copying a flag onto other ships
- `src/lib/features/ship/FlagCard.svelte` — adds the `Apply to other ships…` button next to `Hoist this one` / `Strike this flag`
- `src/lib/state/ui.svelte.js` — extends `openDialog` union with `'shore-leave'` and `'apply-flag'`; adds `applyFlagSource` context pointer + `openApplyFlagDialog(shipId, flagId)` helper

Plumbing in scope (state, persistence, derivations — not UI surfaces):

- `src/lib/domain/derivations.js` — `composeShoreLeaveDeltas` / `composeShoreLeaveSummary` pure helpers; `actionCategory` falls through to `'refit'` for the new `workspace.shore-leave` / `workspace.ship-order` / `workspace.flagApply` / `ship.shore-leave` kinds (intentional — fleet management = between-fight maintenance)
- `src/lib/state/workspace.svelte.js` — `applyShoreLeave(shipIds, deltas, options, source)`, `setShipOrder(newOrder)`, `moveShipUp(shipId)`, `moveShipDown(shipId)`, `applyFlagToShips(sourceShipId, sourceFlagId, targetShipIds, options)` mutators. Multi-ship mutators (Shore Leave + Flag Templates) commit once for the workspace-level undo entry, then `appendShipLog` per affected ship for narrative detail in each Captain's Log

## Findings by Heuristic

### 1. Visibility of System Status — PASS

- [x] `ShoreLeaveDialog` renders a live two-tier preview: a workspace-level summary line ("Shore leave on Lassie, Black Pearl, Interceptor (Tortuga).") plus a bulleted per-ship breakdown ("Shore leave on Lassie (Tortuga): +5 hull, +2 grub.") computed from `composeShoreLeaveSummary` — the same helper `applyShoreLeave` uses to write the actual log lines. The preview pane is wrapped in `aria-live="polite"` so each delta change re-announces (`ShoreLeaveDialog.svelte:312`–`:329`). Same workspace-summary threshold rule the mutator uses (≤3 ships → name list, 4+ → "across N ships") is mirrored verbatim at `:132`–`:140` and `workspace.svelte.js:1899`–`:1902`, so dialog and log are guaranteed to agree.
- [x] `ApplyFlagDialog` shows a single live preview line ("Will copy 'The Black Spear' from Lassie to 2 ships.") in the same `aria-live="polite"` region (`ApplyFlagDialog.svelte:272`–`:285`), and a per-row clash hint (e.g. "Already flies a banner with this name") on each disabled target row at `:241`–`:246`, so the user can read both intent and exclusions in one glance. When every other ship clashes, an amber caption explains the empty selection state ("Every other ship already flies a banner with this name. Strike or rename one to clear a slot.", `:251`–`:255`).
- [x] Both dialogs gate the Apply trigger on a derived `willCommit` flag that requires both at-least-one-target AND at-least-one-meaningful-delta (`ShoreLeaveDialog.svelte:130` and `:336`; `ApplyFlagDialog.svelte:95` and `:292`). The disabled state pairs with positive empty-state copy in the preview pane ("Select at least one ship to begin.", "No effect on the selected ships yet — set hull, supplies, or scene clear.", "Pick one or more eligible ships to copy this banner to.") so the user gets a verbal explanation rather than a silently-disabled button.
- [x] Each fleet-rail card surfaces its position context implicitly: the up arrow disables on the head ship (`ShipCard.svelte:89`), the down arrow disables on the tail (`:101`), and both arrows show a position-aware `title` that reads either "Already at the head of the line." / "Already at the tail of the line." (the disabled case) or "Sail Lassie ahead one slot." / "Sail Lassie astern one slot." (the actionable case) at `:91` and `:103`. A keyboard user tabbing through the rail can hear the position state from screen-reader announcements without leaving the card.
- [x] `Shore leave` toolbar button reflects the workspace state explicitly: disabled when `shipCount === 0` with a tooltip that reads "Load at least one ship to set a shore leave.", flipping to "Refit hull, supplies, and scene chips across selected ships in one undoable step." once a ship is loaded (`WorkspaceTopBar.svelte:50`–`:53`). Same tooltip-toggle pattern v0.8 applied to `Print fleet`.
- [x] Each per-ship row in `ShoreLeaveDialog` shows the live hull state, ship size, and a count of scene tags ("Lassie · Lugger · Hull 14/18 · 2 scene tags", `:220`–`:227`) so the user can assess what each ship will get out of this shore leave at decision time, not only after the apply.
- [x] Toast feedback after `applyFlagToShips` distinguishes three outcomes — pure success ("Flag 'The Black Spear' applied — Copied to 2 ships."), mixed ("…; skipped 1 already flying it."), and warning ("Nothing applied — Every selected ship already flies a flag named 'The Black Spear'.") — so the user knows whether the workspace actually changed without scrolling down to the Activity Log (`ApplyFlagDialog.svelte:142`–`:163`). Toast component already routes warning-class messages through `role="status"` / `aria-live="polite"` and error-class through `role="alert"` / `aria-live="assertive"` (`Toast.svelte:21`–`:22`).

### 2. Match Between System and the Real World — PASS

The v0.9 copy keeps the v0.1 tone split: nautical in headlines / prose, neutral in form labels and field names.

- The Shore Leave dialog title is the verb-as-noun a player would use at the table (`ShoreLeaveDialog.svelte:176`); the description tags the consequence in plainspoken language ("Refit selected ships in one motion — restore hull, top up supplies, and optionally clear scene conditions. Commits as a single undoable step.", `:177`).
- Field labels are neutral nouns ("Hull", "Grub", "Grog", "Gear", "Source") with concrete examples in the placeholder ("e.g. Tortuga, Port Royal, harbormaster", `:307`) and per-field hint text that names the role ("Clamped at each ship's headroom.", "Added to each ship.", "Optional. Names the refit in the log entries.").
- Sail-order tooltips read in the captain's voice — "Sail Lassie ahead one slot." / "Sail Lassie astern one slot." (`ShipCard.svelte:91`, `:103`) — keeping the metaphor consistent with the rest of the app's "head of the line" / "tail of the line" framing.
- Apply-flag dialog title interpolates the flag name ("Apply 'The Black Spear'", `ApplyFlagDialog.svelte:174`); description spells out the consequence the user might miss otherwise ("Each target gets its own copy — future reputation changes on either flag won't propagate back to the other.", `:178`). That last clause matters: reputation propagation across instances of the *same* flag id is the v0.5 model; v0.9's flag-templates path mints fresh ids and breaks that propagation, so calling it out at the moment of decision avoids a subtle gameplay surprise.
- Toast bodies use plural-aware copy with smart quotes ("Copied to 2 ships." vs "Copied to 1 ship."; `Flag "Name" applied`, `:147`–`:160`).
- Workspace log lines from `applyShoreLeave` ("Shore leave on Lassie, Black Pearl, Interceptor (Tortuga).", `:1899`–`:1902`) and `setShipOrder` / `moveShipUp` / `moveShipDown` ("Sailed Lassie ahead of Black Pearl.", `workspace.svelte.js:433`, `:465`) read like the captain's logbook entries the player would write themselves.
- The "Hoist on each target" checkbox label uses the verb sailors use for raising a flag ("Hoist this banner after copying", `ApplyFlagDialog.svelte:268`); when the flag is *not* hoisted, the help text is explicit about the consequence ("Leave off to add it to the locker without changing what each ship currently flies.", `:260`).

### 3. User Control and Freedom — PASS

- [x] Both dialogs are dismissible via every standard escape: backdrop click (`Dialog.svelte:28`–`:30`), Escape key (handled via the native `oncancel` at `:43`–`:46`), explicit Close button with `aria-label="Close dialog"` (`:61`–`:70`), and Cancel footer button (`ShoreLeaveDialog.svelte:335`, `ApplyFlagDialog.svelte:291`). All four paths run through the same `closeDialog()` helper, which clears `ui.applyFlagSource` so a future open doesn't pick up stale source data (`ui.svelte.js:89`–`:94`).
- [x] Both apply flows use the established two-click confirm pattern (`ShoreLeaveDialog.svelte:336`–`:344`, `ApplyFlagDialog.svelte:292`–`:300`) borrowed from `DamageComposer` / `RepairComposer` / `ScenePanel`. First click flips to `confirming = true` and re-labels the primary button ("Confirm: apply shore leave" / "Confirm: apply flag") with a paired "Back" affordance; values reset only on a successful Apply or an explicit Cancel (`:142`–`:165`, `:122`–`:165`). If the underlying state changes such that `willCommit` flips false while the confirm is pending, an `$effect` collapses confirming back to false (`ShoreLeaveDialog.svelte:169`–`:171`, `ApplyFlagDialog.svelte:169`–`:171`).
- [x] **Single undo step for the whole multi-ship action.** `applyShoreLeave` and `applyFlagToShips` each fire exactly one `commit({ kind: 'workspace.shore-leave' | 'workspace.flagApply', summary }, () => { …per-ship mutations… })` so a single Cmd+Z reverses the entire batch (`workspace.svelte.js:1904`–`:1923` and `:2311`–`:2325`). The per-ship `appendShipLog` calls that follow (`:1931`–`:1953`, `:2330`–`:2345`) ride along in autosave but are intentionally outside the undo stack — a deliberate trade documented in both mutators' headers as the "redo-gap caveat".
- [x] **Sail Order is two button-clicks per swap with no drag-modal commitment.** A captain who wants to rearrange three ships does `up, up, down` instead of dragging-and-dropping at pixel precision (`ShipCard.svelte:85`–`:108`). Each click commits one swap as one undo step — three undos to rewind three moves, one undo per move. The `setShipOrder` permutation primitive is exposed (`workspace.svelte.js:373`) but not wired to a UI yet, so a future drag-and-drop slot can drop in without a state-shape rewrite.
- [x] **Reorder sibling structure preserves accessibility.** v0.8's `ShipCard` was a single `<button>` containing the focus action only; v0.9 wraps the focus button and the reorder controls as two siblings inside a `<div>` (`ShipCard.svelte:35`–`:110`), so `<button>`-inside-`<button>` HTML never happens and each interactive region announces independently. The reorder column is grouped via `role="group"` with an `aria-label="Reorder Lassie"` (`:80`–`:83`).
- [x] Apply-flag dialog defaults to *no* targets selected (`ApplyFlagDialog.svelte:51`, reset on open at `:59`). This is a deliberate divergence from `ShoreLeaveDialog`'s pre-select-all default (`ShoreLeaveDialog.svelte:55`) — copying a banner is usually targeted at one or two specific hulls, while shore leave is usually fleet-wide. Forcing the choice on flag-copy keeps a hasty Apply from accidentally flying the same colors across the entire fleet. Documented in `ApplyFlagDialog.svelte:14`–`:17`.
- [x] Multi-select state is dialog-local; closing without applying loses no committed state. Both dialogs reset `selectedIds` and the entered deltas / options in a closed→open transition `$effect` (`ShoreLeaveDialog.svelte:51`–`:65`, `ApplyFlagDialog.svelte:55`–`:64`) using a plain-JS sentinel (not `$state`) for `prevOpen` so writing it doesn't re-trigger the effect — clean reset, no stale selection.

- Buttons inside the dialogs and in the toolbar use the shared `<Button>` component with the standard variant set (`primary`, `ghost`, `danger`); two-click confirm uses the same primary→primary swap with an inline ghost back-button as the v0.8 composers and the v0.7 End-scene affordance.
- The new `Shore leave` toolbar button uses the same `<Button variant="ghost">` + leading-14×14-icon pattern as `Print fleet` (v0.8) and `Save All` (v0.4.1), and slots into the same toolbar row at the same nesting depth (`WorkspaceTopBar.svelte:47`–`:59`). The icon (a barrel/cargo silhouette) keeps the established stroke-width + `currentColor` style of the other toolbar icons.
- `ShoreLeaveDialog`'s number steppers reuse `<NumberStepper>` with the existing `tone` system (`tone={hull > 0 ? 'sea' : 'neutral'}`, `:246`, `:257`, `:267`, `:277`) — same color semantics as `RepairComposer` (entered hull repair = sea/blue, supply add = sea, idle = neutral). A user who learned the tone semantics from the damage / repair composers carries them over.
- Multi-select rows in both dialogs use a `<label>` containing a checkbox + truncated text block, `cursor-pointer` on the row, hover state via `hover:bg-surface-100` (`ShoreLeaveDialog.svelte:212`–`:229`, `ApplyFlagDialog.svelte:226`–`:248`). Disabled rows use `opacity-60` + `cursor-not-allowed` (`ApplyFlagDialog.svelte:227`–`:229`) — same disabled-row treatment v0.5's `FlagConflictsDialog` already uses for unresolved conflicts.
- The "Select all" / "Select none" actionable-link pair lives at the same position in both dialogs (top-right of the `Ships` / `Targets` legend, `ShoreLeaveDialog.svelte:188`–`:209`, `ApplyFlagDialog.svelte:200`–`:222`), with the same `text-sea-700 hover:underline` styling and `disabled:` empty-state. `ApplyFlagDialog`'s variant correctly reads "Select all eligible" (not "Select all", `:211`) because clashing rows are disabled and shouldn't be targeted by the bulk action.
- Workspace log lines for the new actions follow the same "verb + ship + parenthetical source + period" cadence as v0.8's damage / repair lines: "Shore leave on Lassie, Black Pearl, Interceptor (Tortuga).", "Sailed Lassie ahead of Black Pearl.", "Copied flag The Black Spear from Lassie to Black Pearl, Interceptor.". `actionCategory(kind)` falls through to `'refit'` for `workspace.shore-leave` / `workspace.ship-order` / `workspace.flagApply` / `ship.shore-leave` (`derivations.js:576`–`:594`) — intentional: these are between-fight management actions, the same bucket `Refit` already covers in the v0.7 Activity Log filter chip row.
- Per-ship Captain's-Log lines from `applyFlagToShips` choose between "Raised" / "Stowed" verbs based on whether the new flag became the flying flag (`workspace.svelte.js:2334`–`:2335`) — same vocabulary the rest of the flag system uses ("Raise" = hoist, "Strike" = remove, "Lower colors" = unfly).

### 5. Error Prevention — PASS

- [x] **Dedup-by-name in `applyFlagToShips` is enforced both at the UI and at the mutator.** The dialog disables clashing rows at `ApplyFlagDialog.svelte:234` (`disabled={row.clash}`) and the underlying `applyFlagToShips` re-checks at the state layer (`workspace.svelte.js:2271`–`:2278`), so even if a future UI regression let the user check a clash row, the mutator drops it from the apply set and reports it as `skipped`. Comparison key is `name.trim().toLowerCase()` on both sides (`ApplyFlagDialog.svelte:73`, `:79`; `workspace.svelte.js:2266`, `:2274`) so "Black Spear" and "  black spear " count as the same banner without stringly-typed surprises.
- [x] `applyShoreLeave` clamps every numeric delta in three layers: the input is `Math.max(0, Math.floor(Number(...) || 0))` (drops negatives and non-numbers, `:1844`–`:1847`), then per-ship hull is `Math.min(requestedHull, headroom)` (clamps at each ship's `hp.max - hp.current`, `:1868`), then the commit's hull mutation is itself `Math.min(ship.hp.max, ship.hp.current + p.appliedHull)` (`:1913`). A buggy caller, a dialog regression, or a corrupted autosave each get caught by at least one layer.
- [x] `setShipOrder` validates the proposed order in three layers (`workspace.svelte.js:373`–`:408`): (1) array + same length; (2) same set of ship ids — no additions, removals, or duplicates; (3) actually different from the current order. Any failure is a silent no-op so a buggy caller can't corrupt fleet order. Documented in the comment block at `:357`–`:370`.
- [x] `moveShipUp` / `moveShipDown` early-out on `idx <= 0` / `idx >= shipOrder.length - 1` (`workspace.svelte.js:422`, `:454`) AND when the named ship doesn't exist (`:425`, `:457`); the disabled-button states in `ShipCard.svelte` (`:89`, `:101`) mean the click never fires in those cases anyway, but the state-layer guard is the safety net.
- [x] `applyShoreLeave`'s "meaningful per-ship guard" (`:1872`–`:1878`) drops a selected ship from the apply set if it's already at full HP, gets no positive supply additions, and has no scene conditions to clear. This means a fleet-wide Cmd+Apply with one ship at full HP doesn't generate a noise per-ship log entry that says "Shore leave on X: " with no deltas. If *every* ship is in that state the whole call is a no-op (`:1891`).
- [x] `ApplyFlagDialog` clears `selectedIds` on every closed→open transition (`:55`–`:64`), so a user who closed the dialog mid-confirm doesn't come back to find their previous selection still ticked. Same for `ShoreLeaveDialog` (`:51`–`:65`); the difference is that ShoreLeave re-pre-selects every loaded ship after the reset (the common case is fleet-wide), while ApplyFlag deliberately re-clears to nothing (the common case is targeted).
- [x] Source field is `maxlength="80"` on `ShoreLeaveDialog` (`:306`) — same length cap the v0.2 ship-name and v0.8 damage/repair source fields use, so a runaway paste can't blow out the workspace log line or the autosave footprint.
- [x] **Closure correctness.** The `applyFlagToShips` commit closure pre-builds the per-ship flag instances in a local `Map` *before* `commit` is called (`workspace.svelte.js:2289`–`:2300`), then the closure body just iterates and assigns. If the closure ran before this, a re-entrant call could see partial state; pre-building keeps the apply atomic.

- [x] Each row in `ApplyFlagDialog`'s target list shows the live hull state plus, for clashing rows, a "Already flies a banner with this name" hint (`:240`–`:246`). The clash isn't surfaced as a generic "skipped" footer; it's a per-row label so the user can see exactly which ships are excluded and why.
- [x] The Apply-flag dialog title interpolates the flag name (`titleText`, `:174`) so the user sees what they're copying without scrolling back to the source flag card. The description names the source ship the same way (`:178`) so context survives between the click on the flag card and the apply on the dialog.
- [x] Field labels and help text are persistent through the shared `<Field>` component — `Hull / Grub / Grog / Gear / Source` for shore leave; `Hoist on each target` for flag-copy. No placeholder-as-only-label patterns. Help text reads in present tense at the moment of decision ("Clamped at each ship's headroom.", `:239`; "Added to each ship.", `:250`; "Raise the new copy as the flying flag on every target.", `:260`).
- [x] Sail-order arrows live on the same row as the ship card (`ShipCard.svelte:80`–`:109`) — the user doesn't have to remember "where are the reorder controls" or right-click anything; they're a constant of the rail.
- [x] The `Shore leave` toolbar button is always visible in the top bar adjacent to `Save All` and `Print fleet` (`WorkspaceTopBar.svelte:47`–`:62`), so the user doesn't have to remember a menu path. Same recognition-vs-recall play v0.4.1 made for `Save All` and v0.8 made for `Print fleet`.
- [x] The two-tier preview in `ShoreLeaveDialog` (`:312`–`:329`) is the recognition aid for a multi-ship action: the workspace summary line plus the per-ship breakdown reads like "here is exactly what I'm about to do, in the same words the log will use after I do it."

### 7. Flexibility and Efficiency of Use — PASS (was PARTIAL pre-v0.9.1)

The v0.9 surfaces hit the headline efficiency win — a fleet-wide refit is now one Apply away, not N visits to N ship details. The three carry-over gaps from earlier audits closed in the v0.9.1 polish bundle: keyboard shortcut for `Shore leave`, autofocus on `ApplyFlagDialog`, and `Alt+Up` / `Alt+Down` reorder shortcuts on focused fleet-rail cards.

- [x] **The headline win, Shore Leave:** in the smoke test, refitting three ships with +5 hull / +2 grub / +1 grog went from "click each ship in the rail, increment hull stepper 5×, increment grub stepper 2×, increment grog stepper 1×, repeat for ship 2, repeat for ship 3" (~24 clicks across three ship-detail visits, three separate Activity-Log lines per resource) to "click Shore leave, type 5/2/1, click Apply, click Confirm" (~5 clicks total, one workspace-level log line + three per-ship Captain's-Log lines, single Cmd+Z to reverse). That's the slate's stated value.
- [x] **The headline win, Sail Order:** rearranging the rail used to require save-all / drag the JSON / reload — no in-app affordance at all. v0.9 lands one swap per click with one undo per swap (`ShipCard.svelte:85`–`:108`, `workspace.svelte.js:421`–`:474`). Disabled-on-edges states keep the buttons honest, no clamp surprise.
- [x] **The headline win, Flag Templates:** copying The Black Spear onto two more hulls used to be: charter a flag on each ship, copy the name + reputation by hand, upload the same art three times. With `applyFlagToShips`, one click on "Apply to other ships…" → multi-select → Apply → Confirm and three matching banners ride at full reputation parity (and, optionally, are hoisted in the same motion via `raiseOnTargets`).
- [x] **Bulk action plumbing carries forward to a future drag-and-drop.** `setShipOrder(newOrder)` is exposed as the permutation primitive (`workspace.svelte.js:373`); `moveShipUp` / `moveShipDown` are thin wrappers. A future v1.x drag-and-drop UI doesn't need a state-shape change — it just calls `setShipOrder(reorderedArray)` and gets the same single-commit / single-undo entry.
- [x] **F7.1 (CLOSED in v0.9.1):** `Cmd+Shift+L` opens the `Shore leave` dialog. The handler in `Dashboard.svelte` mirrors the existing `Cmd+Shift+S` (Save all) and `Cmd+Shift+P` (Print fleet) chords — same `metaKey || ctrlKey` plumbing, same toast on an empty workspace ("No ships aboard. Load at least one ship before calling shore leave."), same gating so the shortcut never opens a useless dialog. The `Shore leave` toolbar button surfaces the shortcut glyph in its tooltip via `shortcutHint('L', { shift: true })`. The platform helper learned `cmd: false` and arrow-key special-casing in the same patch so non-meta combos (Alt+Up / Alt+Down) get the same glyph treatment.
- [x] **F7.2 (CLOSED in v0.9.1):** `ApplyFlagDialog` autofocuses the first non-disabled target row's checkbox on every closed→open transition. The dialog's existing `prevOpen`-gated `$effect` that resets `selectedIds` / `raiseOnTargets` / `confirming` now also schedules a `queueMicrotask` that reads the first id from `eligibleIds` and calls `.focus()` on the corresponding checkbox; `bind:this={checkboxRefs[row.id]}` on the input keeps the lookup table fresh. If every row is a clash (the empty-eligible case), `eligibleIds` is empty and we leave focus on the dialog default — the existing amber caption already tells the user what to do ("Every other ship already flies a banner with this name…").
- [x] **F7.3 (CLOSED in v0.9.1):** `ShipCard`'s focus button now handles `Alt+ArrowUp` / `Alt+ArrowDown` (with no other modifiers) and routes them to `moveShipUp(ship.id)` / `moveShipDown(ship.id)` when the ship isn't at the corresponding edge. Browser default is preserved when no modifiers are pressed (so a future Tab-replacement rail navigator can still claim modifier-less arrow keys). Tooltip glyphs on the visible Move-up / Move-down buttons advertise the shortcut for discoverability — `Sail Lassie ahead one slot · ⌥↑.` (Apple) / `Sail Lassie ahead one slot · Alt+Up.` (Windows / Linux), with the disabled-edge tooltip ("Already at the head of the line.") staying glyph-free since the shortcut wouldn't fire at the edge anyway.

### 8. Aesthetic and Minimalist Design — PASS

- [x] Both dialogs use the standard `<Dialog>` shell (`size="lg"`, `:179` and `:187`) — no bespoke chrome, no decorative iconography, header carries title + description + close, footer carries the two-or-three-button confirm row. Body sections are separated by `gap-4` (`:186`, `:198`) so dense content doesn't crowd; the per-ship lists are `max-h-48` / `max-h-56` (`:210`, `:224`) so the dialog stays compact even with a 12-ship workspace.
- [x] `ShipCard` reorder column is a vertical stack of two icon-only buttons (`ShipCard.svelte:80`–`:108`) with no text labels — the chevrons + accessible title attributes carry the meaning, the visual cost is one extra column on the right edge of each card. Disabled state is `opacity-30` + `cursor-default` (`:87`, `:99`) — visible but non-shouty.
- [x] The `Shore leave` button uses a small inline SVG icon (a barrel silhouette, `WorkspaceTopBar.svelte:55`–`:57`) at the same 14×14 size as `Save all` and `Print fleet`. Toolbar density stays tight; the new action is a peer to the existing two, not a hero.
