# UX Heuristic Audit — ShipSync v0.5 Combat readiness + flags + journal

**Scope:** the v0.5 delta on top of [`v0.4-ux-audit.md`](./v0.4-ux-audit.md). v0.5 ships (a) `CombatReadinessSection` for live Mettle / Crew / Fires editing with per-round Captain's-Log coalescing, (b) `FlagsSection` with workspace-wide reputation propagation and a conflict banner, and (c) `JournalSection` for per-ship session title + narrative authoring with chapter-break "Close session." The supporting `commit()` was extended to merge sequential same-bucket edits into one undo step and one log line. Findings already addressed at v0.1–v0.4 are referenced rather than re-litigated.
**Date:** 2026-04-26
**Surfaces inventoried:** 3 new feature components + 1 changed integration host + 1 changed plumbing module

Components inventoried:

- `src/lib/features/ship/ShipDetail.svelte` — integrates the three new sections in the order CombatReadiness → Profile → Supplies → Officer → PC → Flags → Journal

Plumbing in scope (state and persistence, not UI surfaces):

- `src/lib/state/workspace.svelte.js` — `commit()` extended with optional `coalesceKey` and a same-key merge path; new mutators `setShipMettleCurrent`, `setShipMettleNotes`, `setShipFires`, `setShipCrewCurrent`, `setShipCrewMax`, `setShipCrewSkeleton`, `addShipFlag`, `setShipFlag` (with workspace-wide reputation propagation), `removeShipFlag`, `setShipFlagFlying`, `setShipFlagArt`, `setSessionEntryFields`, `closeCurrentSession`

## Findings by Heuristic

### 1. Visibility of System Status — PASS (post-v0.5.1)

- [x] Mettle / Crew / Fires steppers carry tone changes at meaningful thresholds (`CombatReadinessSection.svelte:56`, `:59`, `:60`).
- [x] Each combat resource lives in a Field with a live status helpText: Mettle baseline (`:88`), Crew skeleton + per-turn allowance (`:104`), Fires explosion DC (`:121`).
- [x] Flags carry a brass "Flying" badge plus a brass border on the flying entry (`FlagsSection.svelte:147`–`:150`, `:208`–`:214`).
- [x] Journal section header shows an Open / Closed pill with a tooltip stating the open-since or closed-at timestamp (`JournalSection.svelte:128`–`:139`).
- [x] Captain's-Log coalescing now correctly re-renders the summary as it advances. *Discovered during the v0.5 smoke test:* mutating `lastEntry.action.summary` in place did not bust Svelte 5's keyed `{#each}` cache, so the displayed summary stayed pinned to the first commit's value. Fixed in `commit()` by replacing the action object instead of mutating it (`workspace.svelte.js:185`–`:209`); verified in-browser that `Mettle 1 → 3` updates as the chain extends.
- [x] **F1.1** *(resolved in v0.5.1)*: Crew tone topped out at amber even at zero crew, equating "stricken / derelict" with "short-handed but still crewed." Now `crewTone` returns `crimson` first when `crew.current === 0`, falling back to amber on the skeleton check (`CombatReadinessSection.svelte:54`–`:64`). Brings Crew in line with the Fires stepper's amber→crimson escalation.

### 2. Match Between System and the Real World — PASS

The new sections keep the v0.1 tone split: nautical in headlines and empty states, neutral in form labels.

- "Combat Readiness" + the "During the fight" subhead (`CombatReadinessSection.svelte:82`–`:83`) frames the section temporally, distinguishing it from the refit-time "Crew configuration" disclosure (`:164`–`:167`).
- "Colors Aloft" + "Reputation & banners" (`FlagsSection.svelte:101`–`:103`) names the section the way a player would; "Hoist this one" / "Lower colors" / "Strike this flag" are all real sailor verbs (post-v0.5.1 — see F4.2 in heuristic 4 for the verb-swap rationale).
- "Captain's Log" + "Per-ship narrative · saved with the ship file" (`JournalSection.svelte:123`–`:126`) names the artifact in-character and immediately disambiguates against the workspace activity log in the right rail (see F4.1).

### 3. User Control and Freedom — PASS

Every v0.5 mutation goes through `commit()`, so undo / redo from the toolbar (and `⌘Z` / `⌘⇧Z`) reverses any combat-resource step, flag edit, flag add / remove, flying transfer, journal title, narrative, and session close. Coalesced commits collapse to one undo step but retain the original `before` snapshot, so `Mettle 1 → 3` undoes back to `1` in a single press (`workspace.svelte.js:185`–`:209`, exercised by `combat-resource coalescing` tests in `workspace.test.js`).

`closeCurrentSession` is reversible: undo restores `endedAt = null` and reverts any title / narrative patch that rode along (`workspace.test.js` "undo of closeCurrentSession reopens the session and reverts patch fields" test).

### 4. Consistency and Standards — PASS (post-v0.5.1)

- [x] CombatReadiness, Colors Aloft, and Captain's Log all use the shared `surface-card` shell, the same `display text-base text-ink-900 uppercase tracking-wider` headline, and the small `text-[10px] uppercase tracking-wide text-ink-500` subhead pattern (`CombatReadinessSection.svelte:80`–`:84`, `FlagsSection.svelte:98`–`:104`, `JournalSection.svelte:120`–`:140`).
- [x] All numeric fields use `NumberStepper` with the same tone vocabulary (`neutral` / `amber` / `crimson` / `sea`) introduced in v0.4.1.
- [x] Field / RuleTooltip / ImageUpload are reused everywhere — no one-off styling.
- [x] **F4.1** *(resolved in v0.5.1)*: The workspace-wide log was renamed **Activity Log** so "Captain's Log" can mean exactly the per-ship narrative. The right-rail h2 now reads "Activity Log" with a subhead "Workspace edits · undo / redo," the empty state was rewritten ("No wake yet." / "Each change you commit to a ship lists here in order. Undo and redo from the toolbar at any time.") to match the new framing, and the feature folder + component were renamed (`src/lib/features/captainsLog/CaptainsLog.svelte` → `src/lib/features/activityLog/ActivityLog.svelte`). Internal doc comments in `workspace.svelte.js`, `ScenePanel.svelte`, and `JournalSection.svelte` were updated to use "Activity Log" for the global feed and reserve "Captain's Log" for the per-ship narrative editor (`JournalSection.svelte`). Tested in-browser: the right-rail h2 reads "Activity Log," edits commit in real time, the per-ship narrative section keeps its h3 "Captain's Log" and there is no longer any visual collision between the two.
- [x] **F4.2** *(resolved in v0.5.1)*: The two flag-removal verbs originally read backwards — "Strike colors" was the reversible no-data-loss action while "Lower this flag" was the destructive `removeShipFlag` call. Real signaling has it the other way: striking is the heavy / surrender act, lowering is the routine put-it-away act. Verbs are now swapped — "Strike this flag" labels the destructive button and warns "Strike this flag from the ship's locker. … Undo will bring it back." in its tooltip; "Lower colors" labels the reversible top-of-section button and tooltips with "Stop flying this flag — it stays in the locker. Hoist any flag to fly again." Section helpText copy on `FLAG_HINT` and `FLYING_HINT` was updated to follow. We held back on `variant="danger"` for the destructive button — the per-row crimson treatment would dominate every flag entry visually; verbs + tooltip carry the weight at a calmer chrome budget.

### 5. Error Prevention — PASS

- All combat-resource mutators clamp through `Math.max(0, Math.floor(...))` (or against `crew.max`); shrinking `crew.max` clamps `crew.current` and `crew.skeleton` down to match in the same commit (`workspace.svelte.js` `setShipCrewMax`).
- `addShipFlag` falls back to "Untitled flag" if the user-supplied name is blank, and inherits reputation from any other workspace ship that already flies the same id — no orphan zero-rep flags.
- `removeShipFlag` cleans up `flyingId` if the removed flag was the active one, and prunes any unreferenced flag art image to keep `workspace.images` from leaking storage.
- `setSessionEntryFields` trims titles and snaps blank ones back to "Session"; it short-circuits on no-op patches so the undo stack stays clean.
- `closeCurrentSession` is defensively a no-op when `sessionHistory` is empty (the auto-logger normally seeds a fresh open entry, but the guard exists).
- Crew config (max + skeleton mark) is tucked behind a `<details>` disclosure (`CombatReadinessSection.svelte:153`–`:196`) so refit-time controls don't crowd mid-fight controls — Heuristic 5 by way of separation.

### 6. Recognition Rather Than Recall — PASS

- Mettle Field shows `Baseline X.` so the user doesn't have to remember the captain's rank (`CombatReadinessSection.svelte:88`).
- Crew Field shows `Skeleton 7 of 25 · 3 actions/turn.` so both the skeleton mark and the per-turn allowance are visible at decision time (`:104`).
- Fires Field shows `Explosion DC 12.` so the user doesn't have to look up the ship profile (`:121`).
- Flag conflict banner names every disagreeing ship and its current value (`FlagsSection.svelte:113`–`:114`); the user picks rather than recalls.
- Journal Field helpText shows `Started Apr 26, 2026, 7:24 PM · 6 actions logged so far.` (`JournalSection.svelte:148`).
- Past sessions list shows title + start → end window + action count (`:217`–`:228`), enough recognition to scan a multi-session voyage.

### 7. Flexibility and Efficiency of Use — PASS

- Per-round coalescing means a flurry of clicks on Mettle (or Crew, or Fires) within one round resolves to one undo step and one log line, instead of four — power users can fiddle without sandbagging the log (`workspace.svelte.js:185`–`:209`).
- Each `NumberStepper` accepts both buttons and direct numeric entry (`NumberStepper.svelte:72`–`:83`), so a power user can type `0` to wipe Mettle in one keystroke.
- Notes textarea, narrative textarea, and journal title commit on blur — keystrokes don't each push a Captain's-Log line.
- Flag art reuses the same `ImageUpload` component as ship + officer portraits — one upload pattern across the app.

### 8. Aesthetic and Minimalist Design — PASS

- Each new section has exactly one h3 + one subhead and no decorative chrome (`CombatReadinessSection.svelte:80`–`:84`, `FlagsSection.svelte:99`–`:104`, `JournalSection.svelte:121`–`:140`).
- Crew configuration is collapsed by default so the section reads as three controls + a notes box (`CombatReadinessSection.svelte:153`–`:196`).
- Journal past sessions are collapsed by default so a long voyage doesn't push the active session offscreen (`JournalSection.svelte:196`–`:241`).
- Flags do *not* yet collapse; for v0.5 a ship typically flies one or two, but a privateer collecting half a dozen could push the section long. Acceptable at this scope; revisit in v0.6 if the playtest surfaces it.

### 9. Help Users Recognize, Diagnose, and Recover from Errors — PASS

- Flag-reputation conflicts (e.g., after loading two `.shipsync.json` files saved out of sync) surface in an amber `role="alert"` banner that names the disagreement and offers a one-click "Match this ship" resolution per flag (`FlagsSection.svelte:106`–`:134`).
- Closed-session message tells the user what state they're in and what to do next: "This session closed at X. Edit anything on this ship to start the next one." (`JournalSection.svelte:187`–`:190`).
- Image upload errors (size limits, file-read failures) ride through the existing `ImageUpload` component which already handles those cases at v0.1.

### 10. Help and Documentation — PASS

Every input in v0.5 is wrapped in a `RuleTooltip` with a sailor-language hint:

- Combat readiness: METTLE_HINT, METTLE_NOTES_HINT, CREW_HINT, CREW_MAX_HINT, CREW_SKELETON_HINT, FIRES_HINT (`CombatReadinessSection.svelte:66`–`:77`).
- Flags: FLAG_HINT, REP_HINT, FALSE_HINT, PIRATE_HINT, FLYING_HINT (`FlagsSection.svelte:78`–`:87`).
- Journal: TITLE_HINT, NARRATIVE_HINT, CLOSE_HINT (`JournalSection.svelte:112`–`:117`).

Persistence asymmetries are surfaced inline: the journal subhead promises "saved with the ship file" (`JournalSection.svelte:124`–`:126`), and the v0.4.1 "autosave only" subhead on `ScenePanel` already speaks to the other side of that contract.

The empty state "No colors flown. Raise a flag to start tracking its reputation across the fleet." (`FlagsSection.svelte:136`–`:140`) teaches the section's purpose at zero cost.
