## Findings by Heuristic

### 1. Visibility of System Status — PASS

- [x] Async actions show progress indicators
- [x] Loading states for disk-bound operations
- [x] Submit / mutation buttons disable + show progress while in-flight
- [x] Success is confirmed (toast / state change)
- [x] Long-running tasks expose progress beyond "loading"
- [x] Status indicators are reachable to assistive tech

**Evidence**

- `src/lib/features/workspace/AutosaveIndicator.svelte:8` — three-state label ("Saving locally…" / "Local save · HH:mm" / "Local save failed") plus a colored dot, surfaced via `aria-label` so screen readers get the same string sighted users get.
- `src/lib/features/fleet/ShipCard.svelte:35` — "Unsaved" chip on cards whose data has drifted from disk; `title=` tooltip explains what it means.
- `src/lib/features/workspace/LoadShipsButton.svelte:8,65` — `busy` flag disables the button and appends an ellipsis ("Open ship file…") during multi-file parsing.
- `src/lib/ui/Toast.svelte:21` — error toasts use `role="alert"` + `aria-live="assertive"`; non-error toasts use `role="status"` + `aria-live="polite"`.
- `src/routes/Dashboard.svelte:144,146` — the resume banner is a `role="region"` with `aria-label="Resume previous session"`; status flips through `autosaveState` are mirrored into `WorkspaceTopBar`.

**Findings**

- Note (not a defect): the v0.1 shell does no real network I/O, so most "loading" cases collapse to disk reads. The autosave indicator handles every state we currently produce; expand verdict-by-verdict in v0.2 once the profile editor adds longer-running mutations.

### 2. Match Between System and the Real World — PASS

- [x] UI copy avoids internal jargon
- [x] Dates / numbers are locale-aware
- [x] Icons are conventional
- [x] Information ordering matches user mental models (ship card → hull bar → status, not DB order)
- [x] Error / empty-state messages are human language

**Evidence**

- `src/lib/domain/derivations.js:345` — `NauticalCopy` table centralizes all empty-state strings ("No ships at anchor.", "Pick a ship, captain.", etc.) so terminology stays in the player-as-captain mental model.
- `src/routes/Dashboard.svelte:155` — autosave timestamp rendered through `new Date(...).toLocaleString()`, not the raw ISO.
- `src/lib/features/workspace/AutosaveIndicator.svelte:13` — autosave time uses `toLocaleTimeString([], { hour: '2-digit', minute: '2-digit' })` for the at-a-glance display.
- `src/lib/features/captainsLog/CaptainsLog.svelte:27` — log entries render time as `HH:mm`, not ISO.
- `src/lib/features/fleet/ShipCard.svelte:32` — anchor glyph (⚓) marks player-character ships; conventional metaphor for "this is your captain."
- `src/lib/features/fleet/AddShipDialog.svelte:140` — Size / Type / Mobility labels match the rules' vocabulary; help text spells out what defaults are derived from ("Drives default hull, crew, and explosion DC.").

**Findings**

- None blocking. v0.2 should keep `NauticalCopy` the single source for empty-state phrasing as more screens land.

### 3. User Control and Freedom — PASS

- [x] Every modal/sheet/dialog has a Close, Cancel, or back affordance
- [x] Multi-step flows have a way to step back without losing safe state — N/A in v0.1, no multi-step flows
- [x] Destructive actions are reversible (undo) or confirmable
- [x] Forms can be exited without losing entered data, or warn before discard
- [x] Navigation gestures (esc-to-close) behave as expected

**Evidence**

- `src/lib/ui/Dialog.svelte:43,61` — dialog has both an explicit close button and an `oncancel` handler that consumes ESC and routes through the same dismiss path; backdrop click also closes (line 28).
- `src/routes/Dashboard.svelte:163` — resume banner gives both "Start fresh" and "Restore" — neither option is destructive without confirmation, and "Start fresh" only clears the autosave key, not loaded files.
- `src/lib/features/ship/ShipDetail.svelte:17` — `confirmAndRemove()` gates ship removal behind a `window.confirm` and explicitly mentions "Saved files are not deleted." so the user understands the scope.
- `src/lib/state/workspace.svelte.js:23,28` — every mutator is wrapped in `commit(...)` and pushed onto a 100-deep undo stack; `Cmd/Ctrl+Z` and the toolbar buttons surface that.
- `src/lib/features/fleet/AddShipDialog.svelte:53,277` — Cancel button + dialog X both reset the form; closing the dialog discards the in-progress draft.

**Findings**

- Minor (v0.2 candidate): the add-ship dialog discards in-progress fields on close without warning. Acceptable for v0.1 because the form is short and recoverable, but flag it once portrait/traits make the form longer.

### 4. Consistency and Standards — PARTIAL

- [x] Buttons, inputs, typography, spacing, and color usage are uniform
- [x] Primary/secondary/destructive button styles are used consistently
- [x] Terminology is uniform
- [x] Platform conventions respected (native `<dialog>`, `<input type=file>`, etc.)
- [x] Iconography style is consistent (1.25–1.5 stroke, 12–24 viewbox, line style)
- [ ] Navigation patterns match the rest of the app — see "duplicate Phase row" finding

**Evidence**

- `src/lib/ui/Button.svelte:19` — single `Button` primitive with four variants (`primary`, `secondary`, `ghost`, `danger`); used everywhere by import (`AddShipDialog`, `Dashboard`, `ShipDetail`, `LoadedShipsRail`, `LoadShipsButton`, `SaveShipButton`).
- `src/lib/ui/Dialog.svelte:39` — uses native `<dialog>` + `showModal()`/`close()` — matches platform behavior for ESC, focus trap, backdrop.
- `src/lib/features/fleet/AddShipDialog.svelte:140,163,193` — every form row uses the shared `Field` + `NumberStepper`/`<select>` primitives; help text and error rendering both go through `Field`.

**Findings**

- **Duplicate "Phase" row in `ScenePanel`.** `src/lib/features/scene/ScenePanel.svelte:23-36` renders a "Phase" select; `src/lib/features/scene/ScenePanel.svelte:52-55` renders a second "Phase" label that just echoes the same value as plain text. Two cells in a 2×2 grid carry the same label and the same datum.
  - **Recommendation:** delete the second "Phase" cell (lines 52–55) and either leave that grid slot empty or replace it with a different scalar (e.g. "Mode" reflecting `workspace.scene.mode`, which is already shown in the header but would benefit from being editable here).
- **Heuristic 4 caveat (informational):** the focused ship card in the rail uses `aria-current="true"` (`src/lib/features/fleet/ShipCard.svelte:23`); when other lists are added later (e.g. session history) we should keep using `aria-current` rather than inventing a parallel signal.

### 5. Error Prevention — PARTIAL

- [x] Forms validate inputs before submit
- [x] Sensible defaults reduce required input
- [x] Inputs are constrained
- [x] Destructive actions have a confirmation step proportional to risk
- [x] State machines reject invalid transitions
- [ ] Image uploads warn at the right size threshold but don't enforce a fleet-wide budget

**Evidence**

- `src/lib/features/fleet/AddShipDialog.svelte:50,51` — submit gated by `canSubmit` derived from a non-empty trimmed name; the disabled-state on the Charter button is wired through `<Button disabled>` (line 279).
- `src/lib/features/fleet/AddShipDialog.svelte:44-48` — `$effect` keeps `hpMax`, `explosionDC`, `crewMax` in sync with the size dropdown until the user explicitly overrides them; help text on each field tells the user that overriding flips off the auto-fill.
- `src/lib/ui/NumberStepper.svelte:13,17,42,65` — number inputs are clamped to `[min, max]`; +/− buttons disable at limits; non-numeric keystrokes early-return without mutating value.
- `src/lib/features/fleet/AddShipDialog.svelte:62` — `skeleton = max(1, ceil(crewMax / 4))` guarantees a non-zero baseline even with weird inputs.
- `src/lib/ui/ImageUpload.svelte:34,43` — uploads over `IMAGE_HARD_CAP_BYTES` are rejected with an explanatory error toast; uploads over `IMAGE_WARN_BYTES` push a warning toast but proceed.
- `src/lib/features/ship/ShipDetail.svelte:19` — `window.confirm` before removing a ship; copy explicitly notes saved files are untouched.
- `src/lib/persistence/loadFile.js` (read at line 1 — invoked from `LoadShipsButton`) — failed files surface per-file errors to the user instead of silently dropping the whole batch.

**Findings**

- **No fleet-wide image budget.** Each upload is bounded by `IMAGE_WARN_BYTES`/`IMAGE_HARD_CAP_BYTES`, but a workspace with five 4 MB portraits will still blow out the localStorage quota. The autosave fallback strips images and surfaces a follow-up label (`AutosaveIndicator.svelte:31`), but we never warn at the moment of upload.
  - **Recommendation:** when an upload would push `Object.values(workspace.images).reduce(...)` past `IMAGE_WARN_BYTES * 3` (or another tunable), push a warning toast pointing the user at "save your ship file to keep the portrait" before they hit autosave loss. Cheap, additive change in `ImageUpload`.
- **Add-ship name field shows nothing on initial render even though it's required.** `AddShipDialog.svelte:50,126` only surfaces `nameError` once the user has typed and then cleared the field. The disabled button + asterisk are arguably enough, but a one-shot inline hint after first blur would prevent confusion when someone tabs through.
  - **Recommendation:** track a `touched` flag on the name field and show `nameError` after first blur, in addition to the existing length-based gate.

### 6. Recognition Rather Than Recall — PASS

- [x] Field labels are persistent (not placeholder-only)
- [x] Selected filters/options remain visible — N/A, no filtered lists in v0.1
- [x] Recently used / recently viewed items surfaced where relevant — captain's log is exactly this for state edits
- [x] Multi-step flows show current step — N/A
- [x] Contextual hints/help are visible at the moment of decision

**Evidence**

- `src/lib/ui/Field.svelte:11` — every form row has a real `<label for>` plus persistent `helpText` and (when validation fails) `errorText`. No placeholder-only fields.
- `src/lib/features/fleet/AddShipDialog.svelte:140-209` — every numeric field shows whether its current value is auto-derived from size or has been customized ("Auto-filled from medium default. Change to override." vs "Custom value — auto-fill is disabled."). The user never has to remember which fields they've touched.
- `src/lib/features/ship/ShipDetail.svelte:71-93` — the four KPI cards (Hull / Mettle / Crew / Explosion DC) each pair the live value with its derived baseline (damage threshold, baseline mettle, skeleton crew, fires) so the user doesn't have to re-derive it from the rules.
- `src/lib/features/captainsLog/CaptainsLog.svelte:23` — captain's log shows the most recent 20 actions newest-first, each with a `kind` label so the player can scan what changed.
- `src/lib/features/captainsLog/UndoRedoControls.svelte:15,27` — undo/redo buttons surface counts in their `title` ("Undo (3 steps available)"), giving the user a recognition cue for how far back they can go.

**Findings**

- None blocking. Once the profile editor lands in v0.2, keep the "auto-filled vs custom" pattern from `AddShipDialog`.

### 7. Flexibility and Efficiency of Use — PARTIAL

- [x] Keyboard shortcuts exist for frequent actions
- [ ] Keyboard shortcuts are discoverable
- [x] Touch/drag gestures present where appropriate — N/A for v0.1 (no drag reorder yet)
- [x] Saved searches / favorites — N/A (the workspace is the favorites list)
- [ ] Bulk actions exist when single-item actions would be repetitive
- [x] Customization (sort order, layout density, theme) — deferred to a later milestone

**Evidence**

- `src/routes/Dashboard.svelte:67-108` — registers `Cmd/Ctrl+S` (save focused ship), `Cmd/Ctrl+Z` (undo), `Cmd/Ctrl+Shift+Z` / `Cmd/Ctrl+Y` (redo). Save shortcut shows a helpful "Nothing focused to save" toast instead of failing silently when nothing is selected.

**Findings**

- **Shortcuts are completely undiscoverable.** No menu, no `?` overlay, no "Tip:" in the empty state, no keyhint in the buttons themselves.
  - **Recommendation:** add the canonical glyphs (⌘S, ⌘Z, ⌘⇧Z) into the `title=` attributes of `SaveShipButton` and `UndoRedoControls`, and (cheap separate change) wire a `?` keyboard handler to a small modal listing the shortcuts. This is a low-risk, additive change.
- **No bulk save.** A player loading three ships into the workspace must click Save once per ship. Cheap shortcut: a "Save all changed ships" button gated on `Object.keys(lastSavedAtByShipId)` divergence; deferrable to v0.3 since most sessions have ≤2 ships.
  - **Recommendation:** track this against the v0.3 milestone, not v0.1. No code change today.

### 8. Aesthetic and Minimalist Design — PARTIAL

- [x] Screens have a single clear primary action
- [x] No redundant copy — see one finding below
- [x] Visual hierarchy guides attention (tokenized type/weight/color)
- [x] No competing CTAs of equal weight
- [x] Decorative elements don't crowd functional ones

**Evidence**

- `src/lib/features/workspace/WorkspaceTopBar.svelte:43-53` — single `Button variant="primary"` (Add ship); Load is `secondary`; undo/redo are an icon group; autosave is a passive label. No two CTAs compete for attention.
- `src/lib/features/fleet/LoadedShipsRail.svelte:30-39,49-59` — empty-state shows one primary action (Charter), and once ships exist the secondary "Add another ship" lives in the footer rail rather than competing visually.
- `src/lib/ui/EmptyState.svelte:11` — empty states are intentionally small (icon + title + body + optional single action), no marketing decoration.

**Findings**

- **Duplicate "Phase" row** (also flagged under Heuristic 4): two of the four ScenePanel grid cells are labeled "Phase" and show the same value. Beyond consistency, this also fails minimalism — it doubles signal weight on a single datum without adding info. Recommendation as listed under Heuristic 4.
- **`ShipDetail` "Coming aboard in later milestones" panel.** `src/lib/features/ship/ShipDetail.svelte:96-104` renders a roadmap stub inside the live detail surface. Useful while v0.1 is the current build, but it doesn't help the user complete a session.
  - **Recommendation:** behind a `__DEV__` flag (or just remove for the next visible-to-others release). Keeping it in v0.1-internal is fine.

### 9. Help Users Recognize, Diagnose, and Recover from Errors — PASS

- [x] Errors are visually distinct (color, icon, position) without relying on color alone
- [x] Error copy is in plain language
- [x] Errors say what went wrong AND what to do next
- [x] Inline errors appear next to the field at fault
- [x] Network/system errors offer retry; permission errors offer the path to fix — N/A (no network in v0.1)

**Evidence**

- `src/lib/ui/Toast.svelte:21,33` — error toasts use `role="alert"` and surface a Dismiss button; the surrounding container also wears the kind via shape + label, not color alone.
- `src/lib/features/workspace/LoadShipsButton.svelte:34-41` — every failed file surfaces its own toast with the file name and the parser's first `severity:'error'` issue ("Could not load X.shipsync.json: schemaVersion 7 not supported"). The user sees what went wrong and which file.
- `src/lib/features/workspace/LoadShipsButton.svelte:22-31` — even the warning path (file loaded but with caveats) shows the first three issue messages so the user knows what was repaired.
- `src/lib/ui/Field.svelte:21` — when a form field fails, the error message uses a red text token but is also placed directly under the field with `role="alert"` — not color-alone.
- `src/lib/features/workspace/AutosaveIndicator.svelte:23-26` — three states are differentiated by both color and label, plus the `imagesStripped` follow-up text "(no images)" tells the user what happened and what they can do (reload the ship file to restore portraits).
- `src/lib/features/fleet/SaveShipButton.svelte:18-23` — save failures surface a toast with the underlying error message, not a silent no-op.

**Findings**

- None blocking. Watch this category as more failure paths land (v0.2: profile editor; v0.4: scene/officers).

### 10. Help and Documentation — PARTIAL

- [x] In-context help (tooltips, info icons, inline hints) exists for non-obvious controls
- [x] Empty states teach users what the screen is for and what to do first
- [ ] Onboarding / first-run guidance exists for novel concepts — partial (empty states do most of this)
- [ ] Links to docs/FAQs are reachable from where the question would arise
- [x] Help content is concise and step-oriented

**Evidence**

- `src/lib/features/fleet/LoadedShipsRail.svelte:35-40` — empty fleet state names what the screen is for ("No ships at anchor.") and points the user at the single next step ("Charter a vessel").
- `src/lib/features/ship/ShipDetail.svelte:39-44` — empty detail state nudges the user to pick or charter a ship.
- `src/lib/features/captainsLog/CaptainsLog.svelte:17-21` — empty captain's log explains why nothing is there yet.
- `src/lib/features/fleet/AddShipDialog.svelte:115` — dialog description ("Name her, set her bones, and decide whether your character mans the helm.") frames the form's intent before the user reads the labels.
- `src/lib/ui/RuleTooltip.svelte:1` — primitive exists for surfacing rule citations on hover/focus.

**Findings**

- **`RuleTooltip` is unused in v0.1.** `src/lib/ui/RuleTooltip.svelte` is built and ready, but no derived display in `ShipDetail` (Mettle baseline, damage threshold, skeleton crew, explosion DC) uses it yet. Since the whole point of the dashboard is to keep ship-combat rules at hand, this is the highest-leverage missed opportunity in v0.1.
  - **Recommendation:** wrap the four KPI cells in `ShipDetail` with `RuleTooltip` instances citing the rules sections (e.g. baseline-Mettle: "Mettle § Baseline"; damage-threshold: "Combat § Damage threshold"). Additive, no behavior change.
- **Top-level `README.md` is one line.** `README.md:1` is just `# ShipSync`, with no project description, no run instructions, no list of save-file conventions. Users (including future-you) can't bootstrap from the repo.
  - **Recommendation:** flesh out `README.md` with a 1-paragraph description, npm script reference, and a pointer to `docs/audits/` and the TDD. Cheap, scope-bounded change.
- **No keyboard-shortcut help.** Reiterating Heuristic 7's recommendation: a discoverable shortcuts list (e.g. `?` opens a modal or static panel) would resolve both the discoverability gap and a chunk of help-and-docs.
