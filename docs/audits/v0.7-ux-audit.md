# UX Heuristic Audit — ShipSync v0.7 Scene lifecycle + Activity Log filters + Officer notes + Flag-locker memory + Session metadata + Print view

**Scope:** the v0.7 delta on top of [`v0.6-ux-audit.md`](./v0.6-ux-audit.md). v0.7 ships six user-visible additions: (a) an explicit `End scene` action in the right-rail Scene panel that clears every ephemeral scene field plus `boardedBy` on every player ship in one undoable commit; (b) a two-row filter strip on the workspace Activity Log (per-ship chips + a Combat/Crew/Refit/Journal segmented control) with a larger read buffer so tight filters still surface meaningful history; (c) a per-station free-text "Duties & quirks" textarea on every `OfficerCard`; (d) per-ship `localStorage` memory for the v0.6 "Show all flags" disclosure (the F7.1 audit fix from v0.6); (e) three free-text fields on every `SessionEntry` (`sessionDate`, `location`, `encounterName`) editable on both the active and past entries; and (f) a dedicated `?print=<shipId>` print sheet route that hydrates from the local autosave and renders one ship in clean, paper-ready layout. Findings already addressed at v0.1–v0.6 are referenced rather than re-litigated.
**Date:** 2026-04-26
**Surfaces inventoried:** 1 new top-level route + 6 changed feature components + 1 new transient-UI slice

Components inventoried:

- `src/routes/PrintView.svelte` — new, single-ship print sheet (read-only, hydrates from autosave)
- `src/App.svelte` — URL-driven `?print=<shipId>` router that flips to `PrintView`
- `src/lib/features/scene/ScenePanel.svelte` — new End-scene footer with inline two-click confirm
- `src/lib/features/activityLog/ActivityLog.svelte` — ship + category filter strip; expanded read buffer (100) and visible cap (30)
- `src/lib/features/ship/OfficerCard.svelte` — adds the "Duties & quirks" textarea below name/rank/status
- `src/lib/features/ship/JournalSection.svelte` — adds the Played/Location/Encounter trio to the active session editor
- `src/lib/features/ship/PastSessionEditor.svelte` — adds the same trio to closed-session cards
- `src/lib/features/ship/FlagsSection.svelte` — wires the "Show all flags" disclosure to per-ship `localStorage`
- `src/lib/features/ship/ShipDetail.svelte` — adds the "Print sheet" header action that opens the print URL in a fresh tab

Plumbing in scope (state, persistence, derivations — not UI surfaces):

- `src/lib/domain/types.js` — `Officer.notes: string`; `SessionEntry.{sessionDate,location,encounterName}: string`
- `src/lib/domain/derivations.js` — `actionCategory(kind)` + `ACTION_CATEGORIES` / `ACTION_CATEGORY_LABELS`; `makeEmptyOfficers` initializes `notes: ''`
- `src/lib/state/workspace.svelte.js` — `setOfficerNotes`, `endScene`, `setSessionEntryFields` extended with the new metadata keys
- `src/lib/state/ui.svelte.js` — `activityLog: { shipFilter, categoryFilter }` + `setActivityLogShipFilter` / `setActivityLogCategoryFilter` / `clearActivityLogFilters`
- `src/lib/persistence/loadFile.js` — `repairShip` initializes `officer.notes` and coerces missing session-meta fields to `''`

## Findings by Heuristic

### 1. Visibility of System Status — PASS

- [x] The End-scene confirm panel previews exactly what's about to be cleared, computed from live workspace state (`ScenePanel.svelte:42`–`:62`, rendered at `:127`–`:138`): "Resetting: round 4 → 0, phase to idle, wind to N, weather gage cleared, pursuit broken off, 2 sighted ships dismissed, 1 scene condition cleared, boarding cleared on 1 ship." The numbers update in real time while the confirm is open, so an undo-or-edit done in another panel is reflected before the player taps Confirm.
- [x] The End-scene button is disabled when there's nothing to reset (`canEndScene` at `:53`–`:62`, wired into the button at `:118`), so the affordance only lights up when the action is meaningful — no "scene ended" log line for an already-idle scene.
- [x] Activity Log header surfaces both the filtered count and the read-buffer total when filters are active: `{filtered.length} / {actionsAll.length}` (`ActivityLog.svelte:103`–`:108`), and a footer line announces "{N} more entr{y/ies} match — narrow the filter to surface them." when a tight filter spills past the visible cap (`:228`–`:232`).
- [x] Filter chips and the category segmented control both expose `aria-pressed` for the active state (`ActivityLog.svelte:125`, `:136`, `:157`, `:168`) so AT users see the same selected/unselected signal as sighted users see from the inverted ink-900 fill.
- [x] PrintView reports its hydration state explicitly: "Loading sheet…" while the autosave read is in flight (`:160`–`:161`), an amber explanatory banner if no autosave exists ("No local autosave found. Open the dashboard first, load this ship, then print." `:53`–`:54`, rendered at `:156`–`:159`), and a separate amber banner if the URL points at a ship that isn't in the snapshot ("No ship found for id `<id>`. Load this ship in the dashboard first, then come back." `:163`–`:166`).
- [x] PrintView surfaces the open/closed state of the active session as an amber "Open" pill next to its title (`PrintView.svelte:410`–`:412`), matching the pill convention used by the dashboard JournalSection so a print sheet doesn't read differently from the live editor.
- [x] Per-officer "Duties & quirks" notes survive autosave and round-trip through `.shipsync.json` (`workspace.svelte.js:setOfficerNotes`); the print sheet renders them inline below the officer's name (`PrintView.svelte:320`–`:324`), so the player can recognize each station's quirks at the table without diving into the dashboard.

### 2. Match Between System and the Real World — PASS

The new copy keeps the v0.1 tone split: nautical in headlines and prose, neutral in form labels. v0.7 picks up that pattern without regressing it.

- "End scene" reads as plain English; the supporting body copy uses sailor's voice ("Done with the fight? End the scene to clear the round, sighted ships, and any boarding pointers." `ScenePanel.svelte:111`–`:113`), and the verbs in the live preview match how a player would describe the reset out loud — "dismissed", "broken off", "cleared" (`:127`–`:138`, mirroring the mutator's summary text at `workspace.svelte.js:1306`–`:1316`).
- The Activity Log category labels use the four pillars of how a player thinks about the dashboard: Combat / Crew / Refit / Journal (`derivations.js:ACTION_CATEGORY_LABELS`), routed through `actionCategory(kind)` (`derivations.js:576`–`:597`) — `scene.*` kinds and per-ship combat-resource kinds (mettle / fires / crew / hp / boarding) all bucket under Combat; officer + PC kinds bucket under Crew; session.* kinds under Journal; everything else (supplies, profile, conditions, flags, weapons, portrait) under Refit.
- The session-metadata field labels are written as the player would say them: "Played" (real-world date), "Location" (in-fiction setting), "Encounter" (the beat) — with concrete example placeholders that establish the expected register: "e.g. Saturday, March 14" (`JournalSection.svelte:219`), "e.g. Cracked Tooth Bay" (`:234`), "e.g. Raid on the Black Spear" (`:249`). The `helpText` on Played explicitly reframes the field as free-form ("Real-world date this came up at the table. Free text — write it however you'd say it." `:210`) so players don't expect calendar-picker semantics.
- Officer notes are framed as "Duties & quirks" rather than a generic "Notes" field (`OfficerCard.svelte:171`), and the captain's placeholder demonstrates the genre: "Notes for this station." (`:182`–`:184`).
- The print sheet uses player-facing typography conventions: ship name in display font at `text-4xl`, type/size/mobility joined with middle dots ("Sloop · Small · Low mobility"), KPI tiles labeled "Hull / Mettle / Crew / Explosion DC" (`PrintView.svelte:177`–`:230`). Sections use small caps headings ("Particulars", "Bridge Crew", "Supplies", "Colors", "Captain's Log") that match the dashboard's existing heading style.

### 3. User Control and Freedom — PASS

- [x] End-scene is a two-click confirm with an explicit Cancel button in the inline panel (`ScenePanel.svelte:144`) — no modal interruption, no double-tap risk, and the confirm panel itself is not the action; the action only fires when the player taps "Confirm: end scene" (`:145`–`:147`).
- [x] The reset is one undoable commit (`workspace.svelte.js:endScene`, single `commit(...)` call at `:1320`–`:1344`), and the confirm body says so verbatim: "Undo restores everything in one click." (`ScenePanel.svelte:140`–`:141`). The supporting tooltip on the End-scene button reinforces the same point — "Reversible with one Undo." (`:39`).
- [x] If the underlying state is wiped while the confirm panel is open (e.g. the player undoes from the workspace toolbar before tapping Confirm), the panel auto-collapses back to the idle button so the player isn't staring at a stale "Resetting:" preview that would no-op (`ScenePanel.svelte:82`–`:84`).
- [x] Activity Log filters are reversible without leaving the view: a "Clear filters" link surfaces in the filter strip whenever any filter is active (`ActivityLog.svelte:174`–`:182`), and the empty state for "filter matches nothing" provides a one-click escape via "Show everything" (`:194`–`:205`).
- [x] If the selected ship gets removed from the workspace, the ship filter auto-clears so the player isn't staring at a ghost-anchored empty list (`ActivityLog.svelte:66`–`:70`).
- [x] PrintView is read-only — it imports neither `commit()` nor any mutator from `workspace.svelte.js`, only the reactive `workspace` proxy and the autosave reader (`PrintView.svelte:25`–`:26`). The sticky action bar at the top exposes Close (`window.close()`, `:118`–`:120`) and Print (`window.print()`, `:114`–`:116`) and gets stripped from `@media print` (`:455`–`:469`) so it doesn't print on paper.
- [x] Officer notes and session metadata commit on blur and snap back to the canonical value when the field is left without a real change (the `re-anchor` pattern, e.g. `OfficerCard.svelte:42`–`:44`, `JournalSection.svelte:99`, `PastSessionEditor.svelte:55`), so a tab-out from an unchanged field doesn't clutter the Activity Log.

### 4. Consistency and Standards — PASS

The v0.7 surfaces stay consistent with the rest of the app — End-scene uses the shared `<Button>` and `RuleTooltip`, filter chips reuse the established `aria-pressed` pattern, and the print route hydrates through the same autosave snapshot as the rest of the dashboard. One pre-existing inconsistency that the PrintView landing made visible (raw enum strings in the ship-header subtitle) was folded in during this audit pass; one minor finding remains as a deferred opportunity:

- [x] Officer notes textarea matches the established input convention: same border, padding, focus styling, and `maxlength` shape as the JournalSection narrative textarea (`OfficerCard.svelte:175`–`:185` vs. `JournalSection.svelte:260`–`:267`).
- [x] Session-metadata inputs in `JournalSection` and `PastSessionEditor` use the same h-9 / h-8 sizing convention used by the rest of those panels and reuse `Field` for label/help wiring on the active session (`JournalSection.svelte:206`–`:252`).
- [x] End-scene action follows the v0.5.1 "destructive verb where it bites" rule: the affordance is named for what it does ("End scene"), and the Confirm primary button reads "Confirm: end scene" — no euphemism, no over-claiming, the action is never described as destructive because it's reversible by Undo and only resets ephemeral scene state plus boarding pointers.
- [x] Activity Log filter chip styling mirrors the chip treatment used elsewhere (h-6, rounded-full, ink-900 active fill) and the segmented category control reuses the same h-6 / inverted-fill pattern in a single-row inline-flex (`ActivityLog.svelte:113`–`:184`).

**Findings**

- **F4.1 (RESOLVED in v0.7 audit pass):** `ShipDetail.svelte:155` previously rendered the dashboard ship-header subtitle as raw enum strings: `{ship.type || 'Unknown type'} · {ship.size} · {ship.mobility} mobility`. With the v0.4.1 label tables in the codebase, every other surface that prints these enum values routes them through `SHIP_SIZE_LABELS` / `MOBILITY_LABELS` — `ProfileSection.svelte:103` (mobility select), `AddShipDialog.svelte:158`/`:182` (charter dialog), `SceneShipCard.svelte:92`/`:110` (scene ship form), and now `PrintView.svelte:180`–`:181`. So a `small` / `low`-mobility ship read "Small · Low mobility" on the print sheet, in the dropdowns, and in the charter dialog — but read "small · low mobility" in the dashboard header. The PrintView landing made this gap visible because two surfaces now render the same line side-by-side and picked different vocabularies.
  - **Resolution:** `SHIP_SIZE_LABELS` and `MOBILITY_LABELS` are now imported in `ShipDetail.svelte` and the header subtitle reads `{ship.type || 'Unknown type'} · {SHIP_SIZE_LABELS[ship.size] ?? ship.size} · {MOBILITY_LABELS[ship.mobility] ?? ship.mobility} mobility`. Folded in during the audit pass — one-line change, no behavior change. Header now matches the print sheet, dropdowns, and the charter dialog.
- **F4.2 (minor):** PrintView's Print and Close buttons are hand-rolled `<button class="...">` rather than the shared `<Button>` component (`PrintView.svelte:137`–`:151`). This is defensible on a route deliberately stripped of dashboard chrome (the print bar is hidden in `@media print` and a `<Button>` import would carry transitive surface-card / variant tokens that print weirdly on paper), but it does mean a future Button restyle won't propagate here. Lower priority — fix only if Button picks up CSS that would survive `@media print` correctly.
  - **Recommendation:** Defer until the next cross-cutting Button overhaul. Track in a follow-up note rather than a pre-merge fix.

### 5. Error Prevention — PASS

- [x] Two-click confirm on End-scene plus the live "Resetting: …" preview means the player can see the consequences before the action fires (`ScenePanel.svelte:127`–`:138`), and the button is fully disabled when the scene is already idle and no boarding pointers exist (`canEndScene`, `:53`–`:62`).
- [x] Auto-cancel of the confirm if the underlying state is wiped (e.g. via undo from another control) prevents a stale confirm from firing on a now-empty scene (`ScenePanel.svelte:82`–`:84`). The endScene mutator itself is a defensive no-op when `!hadAnyScene && boardedShipIds.length === 0` (`workspace.svelte.js:1302`), so even if the UI guard is bypassed somehow, the mutator won't push a noise commit.
- [x] All the new free-text inputs are length-bounded so a single careless paste can't blow out the save file: officer notes 600 (`OfficerCard.svelte:181`), session date 60 (`JournalSection.svelte:218`, `PastSessionEditor.svelte:133`), location 80 (`JournalSection.svelte:233`, `PastSessionEditor.svelte:141`), encounter 80 (`JournalSection.svelte:248`, `PastSessionEditor.svelte:149`).
- [x] PrintView normalizes the URL parameter at the router boundary: empty-string and missing values both fall through to the dashboard (`App.svelte:13`–`:15`). Inside PrintView, a non-existent ship id renders the amber "No ship found for id" banner instead of crashing on a null `ship` (`PrintView.svelte:64`, `:162`–`:166`).
- [x] When the selected ship is removed from the workspace, the Activity Log ship filter auto-clears (`ActivityLog.svelte:66`–`:70`) — no stranded filter, no empty list referencing a ghost.
- [x] Flag-locker `localStorage` access is wrapped in try/catch on both read and write (`FlagsSection.svelte:81`–`:85`, `:91`–`:96`); a quota-exhausted or disabled-storage browser silently falls back to "preference just won't persist" rather than a thrown exception that would tear down the FlagsSection.

### 6. Recognition Rather Than Recall — PASS

- [x] Active filter state stays visible at all times: filtered/total count in the header pill (`ActivityLog.svelte:103`–`:108`), inverted-fill chip + segmented active treatment, and the "Clear filters" link only appears when something is actually applied — making the active state recognizable rather than something the player has to remember they set.
- [x] The Activity Log entries gained a small color-coded category dot (`ActivityLog.svelte:217`–`:221`, `:87`–`:93`): combat = crimson, crew = brass, journal = deep-700, refit = neutral surface-400. Players can scan the log and recognize what kind of action each row is without reading the `kind` token underneath.
- [x] Officer notes appear in the print sheet directly under each officer's name + status pill (`PrintView.svelte:320`–`:324`), so the player can recognize each station's idiosyncrasies at the table without holding them in working memory.
- [x] Session-metadata trio is rendered persistently on every session card — Played, Location, Encounter are not hidden behind a tooltip, they're three named, labeled inputs side by side with help text (`JournalSection.svelte:206`–`:252`). Past sessions show the same trio above the narrative textarea (`PastSessionEditor.svelte:126`–`:154`), so a closed session presents the same recognition surface as the active one.
- [x] PrintView's session block surfaces the metadata as a single concatenated subhead under each entry's title — `[sessionDate, location, encounterName].filter(Boolean).join(' · ')` (`:417`, `:439`) — so a print sheet that lists ten sessions lets the player recognize which one is which without re-reading the narrative.

### 7. Flexibility and Efficiency of Use — PARTIAL

- [x] Activity Log filters and-together (ship × category), so a player tracking a specific encounter can pull "Salt Wraith × Combat" and see only that ship's combat-relevant edits (`ActivityLog.svelte:50`–`:58`). The read buffer is widened to 100 with a visible cap of 30 (`:39`–`:40`) so a tight filter still surfaces meaningful history.
- [x] FlagsSection disclosure now remembers state per-ship in `localStorage` (`FlagsSection.svelte:69`–`:104`), which closes the F7.1 finding from v0.6 — a privateer with eight flags doesn't have to re-open the locker every visit. The key is namespaced (`shipsync.flagsLocker.open.<shipId>`) so different ships keep independent preferences, and the open state is the sole bit stored (closed = key absent), keeping the keyspace small.
- [x] Print sheet is reachable via a header action on every ship (`ShipDetail.svelte:174`–`:189`) and opens in `_blank` with `noopener` so the dashboard tab stays put; the print sheet itself uses a sticky action bar with explicit Print + Close buttons.
- [x] End-scene is a single button + confirm — no menu drill-down, no settings panel — so the canonical "scene's over, sweep the deck" gesture is one-and-a-bit clicks from anywhere in the dashboard.

**Findings**

- **F7.1 (PARTIAL):** PrintView is single-ship only. A player or GM who wants paper sheets for an entire fleet has to print each ship's tab separately. For a v0.7 MVP and a tactical sheet, this is defensible — but a "Print fleet" affordance (one print job, one ship per printed page using `break-after: page`) would be a natural follow-up if printing becomes a frequent table-prep flow.
  - **Recommendation:** Treat as a deferred opportunity. If/when adopted, route it through a new `?print=fleet` URL that hydrates the autosave and renders the existing single-ship template once per ship in `workspace.shipOrder`, separated by `break-after: page`. No state changes required.
- **F7.2 (PARTIAL):** No keyboard shortcut for End-scene, Clear-filters, or Print. The dashboard does carry shortcuts for Save All (⌘⇧S, v0.4 audit fold-in A2) and undo/redo, but the v0.7 surface adds no new shortcuts. End-scene as a shortcut would be risky (destructive-feeling, single chord), so this is a non-issue. Print as a shortcut might be useful for table prep — Cmd+P routes to the system print dialog already, which is the right answer.
  - **Recommendation:** No action. Leaving destructive-feeling actions explicitly mouse-driven is a deliberate v0.4-era pattern.

### 8. Aesthetic and Minimalist Design — PASS

- [x] End-scene confirm panel uses two short paragraphs but each does a different job: the first lists what's about to change in concrete numbers, the second names what's left untouched plus the reversibility guarantee (`ScenePanel.svelte:127`–`:142`). Trimming either would lose information the player needs at the moment of decision.
- [x] Activity Log filter strip stays compact — single-row chip group + single-row segmented control, both at h-6, both inside a single padded surface-50 strip (`ActivityLog.svelte:111`–`:184`). The strip only renders when there are entries to filter (`#if actionsAll.length > 0`, `:111`), so a fresh workspace doesn't show empty filter affordances.
- [x] Officer notes textarea is positioned subordinate to the name/rank/status row inside `OfficerCard` (`OfficerCard.svelte:170`–`:186`) — players who don't need notes can ignore the field entirely, and the rows-2 default keeps the card from doubling in height.
- [x] Print view is a single-column max-w-3xl layout (`PrintView.svelte:155`); KPI grid uses 4 tiles wide (`:203`–`:230`), Bridge Crew uses 2 columns (`:299`), supplies is a single inline row (`:334`–`:338`). No competing CTAs — Print + Close are the only actions on the page, both in the sticky bar that's hidden on paper.
- [x] PrintView's sectioned headings ("Conditions", "Particulars", "Bridge Crew", "Supplies", "Colors", "Player Character", "Captain's Log") only render when the underlying data exists (`{#if (ship.conditions ?? []).length > 0}` `:238`, `{#if (ship.flags?.flown ?? []).length > 0}` `:341`, `{#if ship.playerCharacter}` `:375`, `{#if activeSession || pastSessions.length > 0}` `:400`), so an unflagged ship without conditions or a PC prints a tighter sheet rather than a sheet of empty headings.

### 9. Help Users Recognize, Diagnose, and Recover from Errors — PASS

- [x] PrintView's two error states use plain language plus a concrete next step. "No local autosave found" tells the player exactly what to do: "Open the dashboard first, load this ship, then print." (`:53`–`:54`). "No ship found for id `<id>`" tells the player how to recover: "Load this ship in the dashboard first, then come back." (`:163`–`:166`). Both are amber-bordered banners, not alarm-red — these are recoverable misuse paths, not crashes.
- [x] If hydration throws (corrupt autosave, pickled JSON, etc.), the catch surfaces the underlying message rather than swallowing it (`PrintView.svelte:59`–`:61`) — the same banner renders with the actual error text so the player can copy-paste it into a bug report.
- [x] Activity Log empty-for-this-filter state is named, not silent: "No entries match this filter yet." with a "Show everything" recovery link (`ActivityLog.svelte:194`–`:205`). The "no actions ever" empty state continues to use the existing `EmptyState` component with nautical copy.
- [x] Flag-locker `localStorage` failures are caught silently and a comment names the trade-off (`FlagsSection.svelte:81`–`:85`, `:91`–`:96`): "preference just won't persist". This is the right choice for a non-essential preference — surfacing a quota error to the player would be louder than the actual cost (re-opening the locker once per visit).
- [x] End-scene's no-op guard at `workspace.svelte.js:1302` is a defensive backstop — if the UI somehow allowed the button to fire on an idle scene with no boarding (e.g. via dev tools or a synthetic event), the mutator quietly returns instead of pushing a meaningless `scene.end` log line.

### 10. Help and Documentation — PASS

- [x] The End-scene affordance carries help at three layers: (1) `END_SCENE_HINT` tooltip on the button itself enumerates exactly what's reset and what's left alone, plus the reversibility guarantee (`ScenePanel.svelte:38`–`:39`); (2) the inline body sentence above the button frames the action ("Done with the fight? End the scene to clear the round, sighted ships, and any boarding pointers." `:111`–`:113`); (3) the confirm panel itself explains the "Persistent ship state … stays. Undo restores everything in one click." (`:139`–`:142`) at the moment the player has to decide.
- [x] Officer notes have a station-aware help text + placeholder pair: helpText explains the field is for inter-session memory ("Free text — anything you want to remember about this officer between sessions." `OfficerCard.svelte:173`), and the captain's placeholder demonstrates the genre at the moment of input.
- [x] Session metadata fields each carry their own helpText: Played names the register ("Real-world date this came up at the table. Free text — write it however you'd say it." `JournalSection.svelte:210`), Location and Encounter ground the player ("Where the ship spent this stretch of voyaging." `:225`, "Optional name for this session's main beat." `:240`), and each placeholder gives a worked example.
- [x] Activity Log's per-action `<span class="text-[10px] text-ink-500 font-mono">{action.kind}</span>` (`ActivityLog.svelte:221`) keeps the underlying mutator name visible to power users without making it the primary content; the human summary is the headline. New users can ignore the kind, debugging users can lean on it.
- [x] PrintView's hydration-state messages double as help: when no ship can be found, the message itself is the help ("Load this ship in the dashboard first, then come back." `:163`–`:166`).
- [x] The v0.4.1 `autosave only` pill on the ScenePanel header (`ScenePanel.svelte:91`–`:98`) continues to be the canonical persistence-asymmetry explainer; v0.7 doesn't regress it, and the End-scene tooltip's "Persistent ship state … is left alone." cross-references the same boundary.
