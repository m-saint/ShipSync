# UX Heuristic Audit — ShipSync v0.3 bridge crew, supplies, and bulk save

**Scope:** the v0.3 delta on top of [`v0.2-ux-audit.md`](./v0.2-ux-audit.md). v0.3 ships (a) the full ten-station officer roster (Captain plus the nine secondary stations), (b) the three supply tracks (Grub / Grog / Gear), (c) a "Save all (N)" bulk action in the top bar, and (d) a retry affordance on the autosave indicator when its local write fails. Findings that already shipped at v0.1 / v0.2 are referenced rather than re-litigated.
**Date:** 2026-04-26
**Surfaces inventoried:** 8 components (3 new, 5 changed) + 3 changed plumbing modules

Components inventoried:

- `src/lib/features/ship/SuppliesSection.svelte` — new
- `src/lib/features/ship/OfficerRoster.svelte` — new (composes 10× the existing `OfficerCard`)
- `src/lib/features/fleet/SaveAllShipsButton.svelte` — new
- `src/lib/features/workspace/AutosaveIndicator.svelte` — added the `Retry` button that appears on `state === 'error'`
- `src/lib/features/workspace/WorkspaceTopBar.svelte` — slots in `SaveAllShipsButton` and pipes `onAutosaveRetry` down to the indicator
- `src/lib/features/ship/ShipDetail.svelte` — replaces the standalone captain card with `OfficerRoster` and inserts `SuppliesSection`
- `src/routes/Dashboard.svelte` — wires `makeAutosaver({ onError })` and the `retryAutosave` callback
- `src/lib/features/ship/OfficerCard.svelte` — unchanged, but now used 10× per ship

Plumbing in scope (state and persistence, not UI surfaces):

- `src/lib/state/workspace.svelte.js:516-542` — new `setShipSupplies` mutator
- `src/lib/persistence/autosave.js:69-93,117-147` — `writeAutosave` now returns `null` on failure and `makeAutosaver` exposes `onError`

## Findings by Heuristic

### 1. Visibility of System Status — PASS

- [x] Async actions show progress indicators
- [x] Loading / "in flight" states for disk-bound operations
- [x] Submit / mutation buttons disable + reflect their effect
- [x] Success is confirmed (toast / state change / log line)
- [x] Long-running tasks expose progress beyond "loading"
- [x] Status indicators are reachable to assistive tech

**Evidence**

- `src/lib/features/fleet/SaveAllShipsButton.svelte:21-34` — the button label, count, disabled state, and tooltip are all derived from `dirtyShips(workspace)`. The user can read the top bar and know exactly how many ships have unsaved edits ("Save all (3)") or that nothing has changed since the last save (button greyed and labelled simply "Save all").
- `src/lib/features/fleet/SaveAllShipsButton.svelte:50-69` — every bulk-save run pushes a paired success / failure toast; the success toast counts the ships actually written and even acknowledges the browser's multi-download permission prompt, so the OS-level chrome does not feel surprising.
- `src/lib/features/workspace/AutosaveIndicator.svelte:9-40` — the indicator now renders four states (`saving`, `error`, idle-with-time, never-saved), each with its own dot color, label, and `aria-label`/`title` body. The `error` state additionally surfaces a `Retry` button (`AutosaveIndicator.svelte:53-62`).
- `src/lib/features/ship/OfficerRoster.svelte:24-44` — the roster header carries a derived `{activeCount} of {STATIONS.length} on station` line, so the user can scan the bridge state without counting cards.
- `src/lib/state/workspace.svelte.js:516-542` — every supply nudge produces a single human log line ("Supplies on Tempest's Reach: grub 0 → 1, gear 2 → 3."), keeping the Captain's Log on the right rail in step with editor activity.
- Smoke test confirmed the chain end-to-end: a Grub `0 → 1` step produced the expected log entry, re-enabled "Save all (1)", and the indicator's idle/saved dot stayed green.

**Findings**

- None blocking. Two small future-polish notes:
  - Supply tracks at zero are a meaningful "bad-day" state per their own help text (`SuppliesSection.svelte:20,26`), but the `NumberStepper` itself doesn't tint or warn at zero. Not a regression and the help text already names the consequence; flagging as a v0.4 polish if Grub/Grog become more central.
  - When `succeeded > 0 && failures.length > 0` (`SaveAllShipsButton.svelte:50-69`), the user gets two toasts in a row (success + error). Acceptable today; revisit if partial-failure becomes common.

### 3. User Control and Freedom — PASS

- [x] Every modal/sheet/dialog has a Close, Cancel, or back affordance — unchanged from v0.1/v0.2
- [x] Multi-step flows have a way to step back without losing safe state — N/A
- [x] Destructive actions are reversible (undo) or confirmable
- [x] Forms can be exited without losing entered data — every commit goes through `commit()` and is undoable
- [x] Recovery actions are non-destructive — see `Retry` evidence

- `src/lib/features/fleet/SaveAllShipsButton.svelte:36-69` — Save All is a non-destructive download; the user keeps every file in their Downloads folder if they change their mind.

**Findings**

- None. v0.3 introduced no new destructive paths and the only new "do something" affordance (Retry) is recovery-flavored.

### 5. Error Prevention — PASS

- [x] Forms validate inputs before submit
- [x] Sensible defaults reduce required input
- [x] Inputs are constrained
- [x] Destructive actions have a confirmation step proportional to risk
- [x] State machines reject invalid transitions

**Evidence**

- `src/lib/state/workspace.svelte.js:516-526` — `setShipSupplies` clamps every track to `≥ 0` and floors fractional input via `Math.max(0, Math.floor(Number(patch[key]) || 0))`. The `if (Object.keys(next).length === 0) return` short-circuit means a no-op patch doesn't burn an undo step.
- `src/lib/features/ship/SuppliesSection.svelte:50-52` — the UI-side stepper is bounded to `min={0} max={999} step={1}`. A user can't paste a negative or non-integer value into Grub.
- `src/lib/features/fleet/SaveAllShipsButton.svelte:23,37` — the button is `disabled` when `count === 0` and the `saveAll` handler short-circuits on the same condition. There is no path for a user to "save nothing" or "save the same ship twice in one click."
- `src/lib/state/workspace.svelte.js:setShipSupplies` returning early on no-op (line 526) — covered by `workspace.test.js:146-160` which verifies a redundant patch does not bump the undo count.

**Findings**

- None blocking. The bounded supply mutator extends the same defensive pattern used by HP / Explosion DC / weapons in v0.2.

### 7. Flexibility and Efficiency of Use — PASS

- [x] Keyboard shortcuts exist for frequent actions
- [x] Keyboard shortcuts are discoverable (glyph in tooltips)
- [x] Touch/drag gestures present where appropriate — N/A
- [x] Saved searches / favorites — N/A

**Evidence**

- `src/lib/features/fleet/SaveAllShipsButton.svelte` and `src/lib/domain/derivations.js:138-183` — the v0.2 finding "no bulk save / multi-edit affordance" is now closed. The button is a single click, derives its eligible set automatically from action / save timestamps, and resets after writing. The v0.2 audit explicitly tracked this as a v0.3 candidate; v0.3 delivers it.
- `src/lib/features/workspace/AutosaveIndicator.svelte:53-62` — the autosave-error retry affordance, also tracked in v0.2 as a v0.3 candidate, is in place. A single tap re-attempts the local write without losing unsaved edits.

Tabbing through the natural DOM order works today, but a `]` / `[` step-through would let a player rotate quickly through stricken stations during cleanup. ### 8. Aesthetic and Minimalist Design — PARTIAL

- [x] Screens have a single clear primary action
- [x] No redundant copy
- [x] Visual hierarchy guides attention
- [x] No competing CTAs of equal weight
- [ ] Decorative elements don't crowd functional ones — see "roster vertical mass" finding

**Evidence**

- `src/lib/features/ship/ShipDetail.svelte:109-205` — the stack still reads top-down: header → KPI strip → Particulars → Stores → Bridge Crew → Player character. One primary CTA (Save) per ship, supported by Remove (`variant="danger"`).
- `src/lib/features/ship/SuppliesSection.svelte:37-58` — three steppers in a `sm:grid-cols-3` row; the section is visually compact relative to Particulars and Bridge Crew.
- `src/lib/features/fleet/SaveAllShipsButton.svelte:73-84` — the bulk-save button is a single icon-plus-label affordance; nothing flashy. The disabled state is a clear "nothing to do" signal rather than a banner or modal.

**Findings**

- **Bridge Crew at full strength is a long block.** `src/lib/features/ship/OfficerRoster.svelte:47-53` renders the captain full-width plus nine more cards in a 1- or 2-column grid. With names, rank steppers, status radio groups, portraits, and help text in each card, a fully crewed ship can scroll for several screenfuls under the KPI strip. Nothing is broken — the cards are individually clean and the captain stays prominent — but it dominates the detail view's vertical real estate, and a player who only wants to bump Grub now has to scroll past Bridge Crew on the way to the Player character section. The audit's checklist item "decorative elements don't crowd functional ones" is the closest fit; the concern here is information density rather than decoration.
  - **Recommendation:** treat as a v0.4 candidate. Cheapest first pass: a "Bridge Crew" collapse/expand toggle that defaults to expanded for the focused ship and remembers the last state per ship; user can collapse it once the bridge is stable. A second option is a "compact" `OfficerCard` variant (no portrait, single-row) selectable from the section header. Either retains the current full editor for the sections the user needs and frees vertical space for Particulars / Stores / Player character.

### 9. Help Users Recognize, Diagnose, and Recover from Errors — PASS

- [x] Errors are visually distinct (color, icon, position) without relying on color alone
- [x] Error copy is in plain language
- [x] Errors say what went wrong AND what to do next
- [x] Inline errors appear next to the field at fault — unchanged from v0.2
- [x] Network/system errors offer retry; permission errors offer the path to fix

**Evidence**

- `src/lib/persistence/autosave.js:69-93` — `writeAutosave` now returns `null` when both the full and image-stripped attempts fail, so the failure surfaces upstream instead of being swallowed by a silent quota error.
- `src/lib/persistence/autosave.js:117-147` — `makeAutosaver` exposes both `onWrite` and `onError`. `Dashboard.svelte` flips `autosaveState` to `'error'` on the latter and to `'idle'` on the former, so the indicator's red dot reflects reality.
- `src/lib/features/workspace/AutosaveIndicator.svelte:32-40,53-62` — the `error` state has a labeled red dot, a `title` body that names the likely cause ("Local autosave write failed — usually a full storage quota."), and a paired `Retry` button so the user has a path forward — not just a status. The button visually distinguishes itself from a top-bar CTA via `text-crimson-700` and a smaller hit area.
- `src/lib/features/fleet/SaveAllShipsButton.svelte:50-69` — bulk save aggregates failures by ship name into the error toast body, so the user knows which ships didn't write.
- `src/lib/persistence/autosave.test.js` — the new tests for `writeAutosave` returning `null` (full + stripped failures, non-quota errors) and for `makeAutosaver` invoking `onError` on failure plus transitioning back to `onWrite` on retry are the canonical truth for the new error pathway.

**Findings**

- None blocking. The v0.2 finding "autosave error state still has no retry affordance" is closed.

## Top Recommendations

These are the changes I'd take into v0.4 polish. None blocks v0.3 from cutting; they are ordered by leverage rather than by section.

| # | Recommendation | Heuristics | Outcome |
|---|----------------|-----------|---------|
| 1 | Add a Bridge Crew collapse toggle (or a compact `OfficerCard` variant) so the ten-station roster doesn't dominate the ship-detail scroll. | 8 | **Tracked for v0.4.** |
| 2 | Add a `⌘⇧S` keyboard shortcut for Save All to pair with the existing `⌘S` per-ship save. | 7 | **Tracked for v0.4.** |
| 3 | Tint Grub / Grog steppers amber when the value is zero, since their own help text names zero as the failure mode. | 1, 6 | **Tracked for v0.4.** |

Verification: `npm test` runs all 138 unit tests green; ESLint clean on the touched files; an interactive smoke test in `http://localhost:5173/?v=v3-smoke` confirmed Bridge Crew renders all ten stations, Supplies persist with a single log line per stepper change, Save All transitions from `Save all (1)` → disabled after write, and a follow-up Grub edit re-lit the bulk save badge.

The v0.3 surface is in its strongest shape so far. Eight of ten heuristics fully PASS; the single PARTIAL (Aesthetic and Minimalist Design) is a density observation, not a regression, and is well-suited to a v0.4 polish pass.
